What the Toolkit Actually Is

matlab/matlab-agentic-toolkit went public on April 3, 2026 — one commit, by MATLAB Fellow Michelle Hirsch, zero pull requests, three open issues. This is not a code library. It is a configuration and documentation package that does two things:

  1. Pre-wires agent platform configs so AI coding agents (Claude Code, GitHub Copilot, Cursor, OpenAI Codex, Sourcegraph Amp, Gemini CLI) automatically connect to a running MATLAB installation via the MCP Core Server.
  2. Delivers curated SKILL.md instruction files that encode MATLAB engineering best practices directly into agent context — telling the agent how to write tests, debug without breaking points, review code quality, and build apps.

The actual MATLAB execution capability lives entirely in the separate matlab/matlab-mcp-core-server binary (closed-source, pre-compiled). The toolkit contains exactly one MATLAB file: printCoverageGaps.m, a utility for displaying code coverage holes from a CoverageResult object.

The right mental model Think of the toolkit as the "system prompt layer" for MATLAB agents — it primes the agent with expert MATLAB knowledge before it ever touches your code. The MCP Core Server is what actually lets the agent run MATLAB. Neither replaces the other.

The Skills Catalog — What's There

The skills-catalog/matlab-core/ directory contains six skills. Each is a SKILL.md file structured as expert instruction for an AI agent — rules, patterns, anti-patterns, and worked examples specific to that domain. They require MATLAB R2020b or later (except Live Scripts, which requires R2025a).

testing

Tools: evaluate_matlab_code · run_matlab_test_file · check_matlab_code

Strict Arrange-Act-Assert pattern, matlab.unittest.TestCase inheritance, qualification level hierarchy (verify/assert/fatal/assume), fixture lifecycle (ClassSetup → MethodSetup → Test → MethodTeardown → ClassTeardown), mocking via matlab.mock.TestCase, parametrized tests, UI testing via matlab.uitest.TestCase, coverage reporting via CodeCoveragePlugin, CI integration via matlab.buildtool.

debugging

Tools: evaluate_matlab_code · check_matlab_code

NEVER use dbstop (blocks eval channel). Use fprintf diagnostics, try/catch with full error context, dbtype for line numbers. Workflow: inspect → instrument → execute via eval → iterate → clean up.

reviewing-code

Tools: evaluate_matlab_code · check_matlab_code · matlab_coding_guidelines (resource)

Naming conventions (lowerCamelCase/PascalCase/UPPER_SNAKE), function quality flags (>6 inputs, >50 lines, nesting depth >3), critical issues (eval/assignin/evalin, missing preallocation, hardcoded paths, builtin shadowing), severity hierarchy.

modernizing-code

Tools: evaluate_matlab_code · check_matlab_code

Deprecated function replacement table: csvread→readmatrix, trainNetwork→trainnet, subplot→tiledlayout, datenum→datetime, guide→appdesigner. Anti-pattern removal: eval, str2num, array growing in loops, global variables, clear all.

building-apps

Tools: evaluate_matlab_code · check_matlab_code

All apps as class-based .m files extending handle (never .mlapp binary files), always use uigridlayout, 12 supported UI components, uihtml for bidirectional MATLAB-JavaScript data exchange, always include delete() method.

creating-live-scripts

Requires R2025a · No MCP tools listed

%[text] syntax for text content, section structure with %%, inline equations with $ $ delimiters (double LaTeX backslashes), standardized metadata appendix, version control compatibility rules.

Plus one setup skill:

matlab-agentic-toolkit-setup (toolkit/ skill)

Claude Code only · Automated onboarding

Five-phase workflow: Discovery (silent info gathering) → Plan (single summary for user confirmation) → Execute (MCP server install + plugin registration) → Verify (communication test) → Report. Persists config to ~/.matlab-agentic-toolkit/config.json.

The Broader MATLAB Agentic Ecosystem

The toolkit is one piece in a rapidly growing set of MathWorks GitHub repos, all published under the github.com/matlab organization since late 2025. Understanding the full picture matters for identifying what's actually missing vs. what already exists elsewhere.

⚙️
matlab/matlab-mcp-core-server
The actual execution layer. A pre-compiled Go binary (closed source) that exposes MATLAB as an MCP server. v0.7.0 (April 2, 2026). 5 tools: detect_matlab_toolboxes, check_matlab_code, evaluate_matlab_code, run_matlab_file, run_matlab_test_file. 2 resources: guidelines://coding, guidelines://plain-text-live-code.
9 releases since Oct 2025 · "personal use only" license restriction
📚
matlab/skills
A separate, standalone skills repo with 6 additional skills: matlab-live-script, matlab-test-creator, matlab-test-execution, matlab-performance-optimizer, matlab-uihtml-app-builder, matlab-digital-filter-design. Installable via /plugin install matlab-skills@matlab-skills in Claude Code.
Different repo from the agentic toolkit — some overlap in purpose
📏
matlab/rules
Modular, referenceable coding rule files: matlab-coding-standards.md, live-script-generation.md, matlab-performance-optimization.md. CC-BY-4.0 licensed. Designed to be @-referenced in any agent. No Simulink rules.
CC-BY-4.0 · No Simulink coverage
🤖
matlab-deep-learning/mcpHTTPClient
MATLAB as an MCP client — calling out to MCP-compatible AI services from within MATLAB. Enables MATLAB to act as an agent orchestrator rather than just an agent tool.
Separate org (matlab-deep-learning) · Complements the toolkit
🏭
matlab/mcp-framework-matlab-production-server
Deploy MATLAB functions as MCP tools on MATLAB Production Server. Requires Compiler SDK, MPS R2022a+, MATLAB R2025b+. Uses prodserver.mcp.build() and prodserver.mcp.deploy(). Currently no SSE/streaming support — transactional connections only.
Enterprise deployment path · Significant toolbox requirements
🔧
matlab-deep-learning/llms-with-tools
MATLAB LLM tool-calling framework. Supports OpenAI, Azure OpenAI, Anthropic, and Ollama backends. Provides openAIFunction, openAIChat, ollamaChat, tool schema definition, and JSON-structured output. The foundational "call LLMs from MATLAB" library.
Cloud + local LLM support · Used in most MATLAB agentic examples
MathWorks' strategy is becoming clear MathWorks is building a layered ecosystem: llms-with-tools (call LLMs from MATLAB) + matlab-mcp-core-server (let agents call MATLAB) + mcpHTTPClient (MATLAB calls other MCP servers) + mcp-framework-matlab-production-server (enterprise deployment) + matlab-agentic-toolkit (knowledge layer / skills). The toolkit is the newest and most surface-level piece of a stack that goes quite deep.

Skill Gaps — Missing Domains

The current toolkit covers 6 MATLAB coding domains. Here is what's absent — domains where an AI agent working with MATLAB has no expert guidance baked in and will generate lower-quality or incorrect code.

The Largest Missing Domain: Simulink

Simulink is MathWorks' primary product and the tool most MATLAB engineers use for serious engineering work. There is no Simulink skill in the toolkit, no Simulink section in matlab/rules, and the MCP Core Server exposes no Simulink-specific tools. An AI agent coding with Simulink today has no MATLAB-expert knowledge layer — it has to guess at best practices from training data alone.

A Simulink skill should cover at minimum:

  • The correct simulation API: out = sim(in) where in is a Simulink.SimulationInput object (not load_system + set_param + sim)
  • Input signal construction: timeseries objects combined into Simulink.SimulationData.Dataset
  • Data logging: accessing logsout, plotting via timeseries commands
  • Parallel parameter sweeps: parsim() with array of SimulationInput objects
  • Model advisor: Simulink.ModelAdvisor.run() for automated quality checks
  • Bus object patterns and signal routing standards
  • Callback conventions (model callbacks vs. block callbacks)
  • Subsystem vs. referenced model patterns
  • Configuration set management: Simulink.ConfigSet

Other Missing Skill Domains

Signal Processing Missing

DSP System Toolbox and Signal Processing Toolbox are widely used. No skill for filter design workflows (designfilt, filterDesigner), spectral analysis (pspectrum, spectrogram), streaming signal processing via dsp.System objects, or the correct use of sample-rate-aware functions. Note: matlab/skills has a separate matlab-digital-filter-design skill, but it's not integrated into the agentic toolkit.

Control Systems Design Missing

Control System Toolbox is one of MATLAB's most-used toolboxes. No skill for tf()/ss() model creation, pidtune(), lqr()/lqe(), bode()/nyquist()/rlocus() analysis, feedback() loop construction, or how agents should work with Simulink Control Design workflows.

Data Science and Machine Learning Missing

No skill for Statistics and Machine Learning Toolbox patterns: fitcsvm(), fitrensemble(), crossval(), confusionchart(), feature engineering with array2table(), or the correct MATLAB data science pipeline. Deep Learning Toolbox: trainingOptions(), trainnet() (the replacement for trainNetwork), layer graph construction, transfer learning patterns.

Optimization Missing

Optimization Toolbox: fmincon() option structures, ga()/particleswarm() for global optimization, problem-based optimization with optimproblem()/optimvar(), solver selection guidance. No skill despite optimization being a core use case in engineering simulation workflows.

Performance Profiling and Optimization Partial

matlab/skills has a matlab-performance-optimizer skill and matlab/rules has a performance rules file, but neither is integrated into the agentic toolkit. No guidance on using the MATLAB Profiler (profile on/off/viewer), identifying bottlenecks, vectorization patterns, preallocation strategies, or tall arrays for out-of-memory data.

Build Automation and CI/CD Missing

MATLAB Build Tool (buildfile.m, matlab.buildtool.tasks.*) is the modern way to define CI pipelines. No skill for build task definition, dependency graphs between tasks, or integrating MATLAB builds with GitHub Actions, Jenkins, or GitLab CI. The testing skill references buildfile.m minimally but doesn't teach it.

Code Generation (MATLAB Coder / Simulink Coder) Missing

Generating C/C++ from MATLAB or Simulink for embedded targets is a major workflow. No skill for codegen configuration, supported/unsupported function lists, MEX compilation, coder configuration objects, or fixing "code generation not supported" errors that agents will commonly generate.

Package and Project Management Missing

MATLAB Projects (introduced R2019a) and namespace packages (+namespace/ directory structure) have no skill. No guidance on matlab.project.Project API, project startup/shutdown files, MATLAB Add-On packaging (prj files), or path management patterns that don't rely on addpath.

Data Import/Export Missing

No skill for reading diverse engineering data: time-series data from instrumentation (timetable), large datasets (datastores, tall arrays), HDF5/h5read, MAT file versioning, binary protocol parsing. Agents default to csvread/xlsread (deprecated) without this guidance.

Parallel and GPU Computing Missing

Parallel Computing Toolbox: parfor constraints (no shared state, no break statements), parpool management, parfeval for async work, spmd blocks, gpuArray patterns. Agents generate incorrect parfor loops without domain knowledge.

Documentation Generation Missing

No skill for writing H1 lines, function help text that help()/doc() parse correctly, or mlx documentation patterns. Agents generate inconsistent or malformed help text without explicit guidance.

The Simulink Problem

Simulink's absence from the toolkit is not just a skill gap — it's the most important missing piece, and it's harder to fix than adding a SKILL.md file.

Why Simulink Is Architecturally Different from MATLAB

The current MCP tools work well for MATLAB code because MATLAB code is text. A file is a .m file. You can read it, write it, check it with checkcode, and run it with evaluate_matlab_code.

Simulink models are different:

  • Binary format: .slx files are ZIP archives. An agent cannot read a Simulink model as text (though .mdl is text, it's legacy).
  • GUI-heavy creation: Models are traditionally built through a graphical block editor. Programmatic model construction (using add_block, add_line, set_param) is powerful but verbose and error-prone.
  • Long execution time: A Simulink simulation can run for seconds to hours. Synchronous tool calls block the agent context window for the full duration.
  • License complexity: Running any Simulink model requires checking out a Simulink license, potentially plus additional blockset licenses. License checkout failures are a common runtime error that agents must handle.
  • State mutation: Opening a Simulink model changes MATLAB's state in ways that evaluate_matlab_code doesn't fully account for. Agents can accidentally leave models open, consuming memory and licenses.

Simulink-Specific Failure Modes for Agents

Failure ModeMATLAB ErrorRequired Agent Response
License checkout failureUnable to check out a Simulink licenseWait and retry with exponential backoff — not fail immediately
Numerical solver divergencesim() completes but output contains Inf/NaNValidate output numerics, not just check for exception
Model crash / dirty stateSimulink corrupts workspaceRun bdclose all + reinitialize before retrying
Memory exhaustionOut of memory mid-runMonitor memory before simulation; reduce logging granularity
Toolbox license cascadeSub-license checkout failure for blocksetCall detect_matlab_toolboxes first; verify all required licenses are available
Async simulation timeoutLong-running simulation exceeds agent timeoutAsync dispatch; poll for completion; resume from checkpoint
The synchronous tool call problem for simulation The MCP protocol is synchronous by design for simple tools. A Simulink simulation that runs for 20 minutes will block the agent's tool call for 20 minutes — keeping the LLM context window open, consuming tokens, and risking timeout. This is an architectural problem that requires either (1) an async/polling model for long-running tools, or (2) a run_simulink_model tool with a configurable timeout that returns a job ID for later polling. Neither pattern exists in the current toolkit.

Enterprise and Production Gaps

The current toolkit is designed for a developer sitting at a workstation with Claude Desktop or Claude Code installed. It is not designed for enterprise deployment. The gap between these two contexts is substantial.

Air-Gapped and On-Premises LLM Support

The primary MATLAB user base — aerospace, defense, automotive, semiconductor — works in environments where external API calls to Anthropic or OpenAI are prohibited. ITAR-controlled programs and classified work cannot send code to cloud APIs. Even in unclassified commercial environments, legal and IT policies often block third-party AI service calls.

MATLAB already supports local LLMs via llms-with-tools (Ollama backend: ollamaChat("llama3.2", Tools=tools)). MathWorks has documented running LLMs on on-premises NVIDIA DGX hardware and connecting via MATLAB. But the matlab-agentic-toolkit does not document or test against any local backend. Model selection (cloud vs. on-prem) is not a first-class concept.

Missing: Air-gapped deployment guide For the core MATLAB user base (aerospace, defense, automotive), a local LLM stack is not optional — it's the only option. The toolkit needs an explicit air-gapped configuration path: Ollama or vLLM backend, local embedding models, self-hosted vector database, no MathWorks telemetry.

Observability and Tracing

The toolkit produces no structured telemetry. An agent run leaves no queryable record of what tools were called, what arguments were passed, what outputs were returned, how long each step took, or why the agent made the decisions it did.

For self-hosted deployments, Langfuse is the strongest path — MIT-licensed, runs on PostgreSQL, full self-host documentation. MATLAB can call Langfuse's REST API via webwrite to emit traces without any Python dependency.

Agent Memory and State Persistence

Every agent session starts from zero context. There is no memory of previous sessions, no retrieval of similar past simulations, no accumulated institutional knowledge. This is a fundamental limitation for engineering workflows where an agent should learn from a project's history.

A MATLAB-native memory architecture could use:

  • Episodic memory: A MATLAB table of past tool calls (timestamp, tool, input, output) persisted to a MAT file — queryable by recency or project
  • Semantic memory: Embeddings of past simulation results stored in a local Qdrant instance — retrieved by semantic similarity to the current task
  • Procedural memory: The skills catalog itself, versioned per project via git

Agent Testing and Evaluation

There is no testing framework for agents in the toolkit. Evaluating whether an agent's behavior has regressed after a skill update or model change requires:

  1. A golden dataset of prompt → expected tool call sequence mappings
  2. Assertions on tool call behavior (not just output text — LLM outputs are non-deterministic)
  3. An LLM-as-judge for semantic correctness evaluation
  4. Integration with MATLAB's matlab.unittest framework for CI pipeline inclusion
  5. Trace replay capability — run recorded agent sessions with mock tool outputs

None of this exists. A team relying on the toolkit has no regression testing for agent behavior.

Human-in-the-Loop and Approval Gates

For high-consequence actions — writing to a model file, triggering a 2-hour simulation, deploying to production — there is no built-in approval mechanism. The agent executes immediately. In aerospace and defense contexts, regulatory frameworks (EASA NPA 2025-07, emerging DO-178C interpretations for AI) explicitly require human approval points for AI-influenced engineering decisions.

Audit Trail for Regulated Environments

Regulatory compliance in aerospace (DO-178C / EASA), automotive (ISO/PAS 8800:2024), and medical devices (ISO 13485) requires an immutable record of every AI-influenced decision. For a MATLAB agent working on safety-critical software, this means capturing:

  • Authenticated user identity (who invoked the agent)
  • Complete prompt and response history (tamper-evident)
  • Every tool call: arguments, outputs, timestamps
  • Human approval/rejection decisions with rationale
  • Data lineage: which model files and parameter sets the agent accessed
  • Artifact hashes: what files the agent created or modified

The current toolkit produces no audit trail whatsoever. This is a disqualifying gap for regulated use cases.

Prioritized Recommendations

Organized by impact and feasibility. Skills and documentation changes are fast; new MCP tools require modifying the closed-source binary; enterprise capabilities require significant architectural work.

Tier 1 — Add Now (Skills / Documentation Only)

Simulink Modeling Skill

The most impactful single addition. Cover the Simulink.SimulationInput API, correct sim() usage, signal construction, data logging, parallel sweeps via parsim(), and failure mode recovery (license checkout, solver divergence, bdclose all cleanup). Add to matlab/rules as simulink-simulation-standards.md for broad agent access.

Control Systems Design Skill

Cover the full Control System Toolbox workflow: model creation (tf/ss/zpk), analysis (bode/nyquist/rlocus/step), synthesis (pidtune/lqr), and Simulink Control Design integration. This is MATLAB's second-most-used toolbox in engineering contexts.

Signal Processing Skill

Merge the existing matlab-digital-filter-design skill from matlab/skills into the agentic toolkit and extend it: designfilt/filterDesigner, pspectrum/spectrogram/periodogram, dsp.System streaming objects for real-time processing, sample-rate-aware function selection.

Performance Profiling Skill

Integrate the matlab-performance-optimizer skill from matlab/skills. Add profiler workflow: profile on → run → profile off → profile('info') for structured data. Teach agents to read profiler output, identify hot spots, and apply vectorization, preallocation, and parfor patterns correctly.

Build Automation Skill

Cover buildfile.m task definition, task dependency graphs, matlab.buildtool.tasks.TestTask/CodeIssuesTask/MexTask, and how to integrate MATLAB builds into GitHub Actions / GitLab CI. Agents can help maintain and extend CI pipelines, not just write code.

Air-Gapped Configuration Guide

Add a GETTING_STARTED_AIRGAPPED.md covering: Ollama installation and MATLAB connection, Ollama model selection for code tasks (Qwen2.5-Coder, Deepseek-Coder-V2), disabling telemetry (--disable-telemetry), self-hosting options, and the MathWorks DGX Spark pattern. This unlocks the toolkit for its primary user base.

Tier 2 — High Impact (Requires MCP Server Changes)

run_simulink_model Tool

A dedicated Simulink tool in the MCP Core Server that: accepts a model name and SimulationInput spec as structured parameters, configures a timeout, runs sim(), and returns structured output (logged signals, simulation metadata, warnings). Includes Simulink-specific error classification (license failure, solver divergence, memory error) for typed error handling.

get_workspace_variables Tool

Return a structured list of workspace variables: name, class, size, bytes, type. Enables agents to inspect state before and after tool calls without parsing whos text output. Critical for debugging loops.

get_profiler_data Tool

Run the MATLAB profiler on a function and return structured performance data (function name, call count, total time, self time, source lines). Agents can use this to implement automated performance optimization loops.

Tier 3 — Enterprise (Significant Architectural Work)

Async Tool Execution Model

For long-running simulations: a submit_job / poll_job / get_job_result tool pattern. The agent submits a Simulink simulation, gets a job ID, continues with other work (or waits), and retrieves results when ready. Prevents agent context timeout on 30-minute simulations. Optionally backed by MATLAB Parallel Server for distributed execution.

Structured Audit Logging

Every tool call logged to an append-only structured record: timestamp, user, tool, arguments, output, latency, model version. Optional Langfuse integration via MATLAB REST calls. Hash-signed for tamper evidence. Configurable retention. Enables regulated industry adoption and provides the observability data that enterprise teams require.

Agent Evaluation Framework

A matlab.agenttest framework (analogous to matlab.unittest) for testing agent behavior: golden dataset format, tool call sequence assertions, LLM-as-judge integration for semantic correctness, CI integration via buildfile.m. Enables regression testing when skills or models are updated.

Human-in-the-Loop Approval Gates

A requires_approval flag in skill manifests for high-consequence actions (file writes, simulation runs, deployment). The agent pauses, presents a summary, and waits for explicit user confirmation before proceeding. For regulated environments, approval is logged with the audit trail.
