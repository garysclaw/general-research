Executive Summary

marimo (v0.21.x, acquired by CoreWeave October 2025) is a reactive Python notebook that replaces Jupyter's REPL model with a directed acyclic graph (DAG) of cell dependencies. The fundamental difference: Jupyter doesn't know how cells relate to each other — you run them in whatever order you want, and hidden state accumulates. marimo builds a dependency graph from Python ASTs and guarantees every re-run produces the same result in the same order.

Jupyter achieves language agnosticism through a language-neutral kernel protocol (ZeroMQ messaging). Over 40 language kernels exist — Julia, R, Haskell, Scala, Go, Rust. marimo is Python-only by architectural design, not oversight. The reactive DAG is built by parsing Python ASTs; there is no equivalent path for R or Julia. The marimo team has explicitly closed R support as "not planned" (GitHub Issue #6620). SQL is the one non-Python language with first-class support, via DuckDB.

The closest analogue to marimo in the Julia ecosystem is Pluto.jl — a reactive, DAG-based Julia notebook that directly inspired marimo's design. Multi-language reactive notebooks don't exist yet as a single product. The gap is architectural, not just a matter of effort.

What is marimo? The Reactive Model

marimo was publicly launched in January 2024 (second-highest Python ShowHN of all time), incubated at Stanford's SLAC National Accelerator Laboratory. It was acquired by CoreWeave on October 30, 2025. It is open-source, Apache 2.0-licensed, with 18,000+ GitHub stars and 180,000+ weekly PyPI downloads as of April 2026.

The DAG Execution Model

Every cell in a marimo notebook is a node in a dependency graph. Edges are defined by variable references: if cell B reads a variable defined in cell A, there is an edge A → B. marimo builds this graph at edit time using static analysis — running ast.parse() on every cell and extracting variable definitions and references without executing any code.

# Cell 1 — defines `data` data = load_csv("measurements.csv") # Cell 2 — defines `filtered` (reads `data`) filtered = data[data["value"] > threshold.value] # Cell 3 — reads `filtered`, defines `chart` chart = mo.ui.altair_chart(filtered.plot()) ───────────────────────────────────────────── DAG edges: Cell 1 → Cell 2 → Cell 3 When `threshold` slider changes: → Cell 2 re-runs (reads `threshold` + `data`) → Cell 3 re-runs (reads `filtered`) → Cell 1 does NOT re-run (no dependency on `threshold`)

This produces two guarantees Jupyter cannot make:

  1. Deterministic execution order. Cells always run in DAG-topological order, regardless of how the developer placed them on screen.
  2. No hidden state. Delete a cell and its variables are immediately removed from memory. The code visible on screen is always a complete description of program state.

Key Constraints (What marimo Forbids)

The DAG model requires rules that Jupyter never enforces:

  • No variable redefinition across cells. If x = 1 appears in two different cells, marimo raises a MultipleDefinitionError. Each global variable must have exactly one cell as its definition.
  • No cycles. If cell A references cell B and cell B references cell A, marimo raises a CycleError. A DAG, by definition, has no cycles.
  • Mutations are not tracked. my_list.append(x) modifies an object without creating a new binding; the static analyzer does not see this as a definition change. Only global variable reassignment (my_list = [...]) triggers reactivity.
  • No IPython magic commands. %timeit, %matplotlib inline, !pip install — none of these work. marimo provides its own alternatives (mo.cli, built-in package manager).

Storage Format: Pure Python, Not JSON

Jupyter stores notebooks as .ipynb — a JSON file containing cell source, outputs (including base64-encoded images), execution counts, and kernel metadata. This format is hostile to version control: git diff produces unreadable JSON noise, merge conflicts invalidate the JSON structure, and execution counts change on every run.

marimo stores notebooks as plain .py files. Each cell is a Python function decorated with @app.cell:

@app.cell
def __(mo):
    slider = mo.ui.slider(0, 100, label="Threshold")
    return slider,

@app.cell
def __(data, slider):
    filtered = data[data["value"] > slider.value]
    return filtered,

This file is executable as a Python script (python notebook.py), importable as a Python module, lintable with Ruff/Black, and produces clean, human-readable git diffs.

Trade-off on outputs Because marimo does not embed outputs in the file, a shared notebook must be re-executed to see results. This is the intended workflow. For notebooks intended as static reports with pre-computed results (e.g., a snapshot shared with a non-technical stakeholder), Jupyter's output-embedded format has a genuine advantage.

Two Operational Modes

Edit Mode — marimo edit notebook.py

Full development environment: reactive cell editor, variable explorer, dependency graph viewer, built-in AI assistant, package manager. Changes trigger automatic re-execution of downstream cells.

App Mode — marimo run notebook.py

Strips all code, exposes only outputs and mo.ui widgets to end users. The notebook becomes a read-only web application — no Voilà, Streamlit, or Dash required.

WASM / Browser Execution

marimo can run entirely in the browser via Pyodide (WebAssembly port of CPython). Export a notebook as a self-contained HTML file with marimo export html-wasm notebook.py -o dist/ and deploy to any static host — GitHub Pages, Cloudflare Pages, S3 — with no Python server required. Cloudflare's notebooks.cloudflare.com is built on marimo.

WASM limitations: 2 GB memory cap, no multiprocessing, no compiled extension packages not in the Pyodide distribution, must be served over HTTP (not file://).

SQL as a First-Class Language

SQL is the one non-Python language with native marimo support. SQL cells are powered by DuckDB and support querying DataFrames, local files, and remote databases (PostgreSQL, MySQL, Snowflake, BigQuery, MotherDuck). The output of a SQL cell is a DataFrame that participates in the Python DAG like any other variable.

# SQL cell (selected from cell type dropdown)
SELECT station_id, AVG(temperature) as avg_temp
FROM monthly_readings
WHERE year = 2025
GROUP BY station_id
ORDER BY avg_temp DESC

# The result is automatically bound to a Python variable
# named after the cell — other cells can reference it directly

Jupyter's Architecture and Language Agnosticism

Jupyter's ability to run 40+ languages is not a feature that was added later — it was an architectural decision made at the project's founding. The insight: separate all language-specific logic into a kernel process, and communicate between frontend and kernel via a language-neutral protocol.

The Kernel Protocol (ZeroMQ)

Every Jupyter kernel implements the Jupyter Messaging Protocol — a specification for JSON-formatted messages transmitted over ZeroMQ (ØMQ) sockets. The frontend and kernel are fully independent processes. They share nothing except this protocol.

┌─────────────────────────────────────────────────────────┐ │ JupyterLab Frontend │ │ (TypeScript/React, runs in browser) │ └──────────────┬─────────────────────────────┬────────────┘ │ ZeroMQ messaging │ │ (5 channels: Shell, │ │ IOPub, stdin, Control, │ │ Heartbeat) │ ┌──────────────▼──────────────┐ ┌──────────▼────────────┐ │ Python Kernel (IPykernel) │ │ Julia Kernel (IJulia) │ │ executes Python code │ │ executes Julia code │ │ sends outputs over IOPub │ │ sends outputs │ └─────────────────────────────┘ └───────────────────────┘ Any language that can implement this protocol gets full Jupyter support: completions, rich output, widgets, debugging.

Current Jupyter Ecosystem

  • JupyterLab 4.x — The primary IDE interface. Multi-panel layout, full extension ecosystem, debugger, LSP integration, Git support.
  • Jupyter Notebook v7 — Rebuilt on JupyterLab internals. Single-document view for users who prefer it. Current stable: v7.5.5 (April 2026).
  • JupyterHub — Multi-user server with pluggable authentication (OAuth, LDAP, PAM), environment spawning, per-user instances. The standard enterprise deployment layer.
  • Cloud integrations — Google Colab, AWS SageMaker, Azure ML, Databricks, Deepnote all use the Jupyter protocol natively.
  • Quarto — A document system that uses Jupyter as an execution engine, inheriting all kernel support (Python, R, Julia in one document).
Why Jupyter's kernel protocol was visionary By placing all language-specific logic in an isolated process communicating over a documented protocol, the Jupyter team made it possible for any language community to add Jupyter support independently. The Python team didn't need to build R support — the R community built IRkernel. This is why Jupyter achieved 40+ language kernels without the core team writing a single line of R, Julia, or Haskell.

Reproducibility: The Numbers

Two studies quantify Jupyter's reproducibility problem:

  • Pimentel et al. (2019), NYU / Federal Fluminense University: Of ~1 million Jupyter notebooks on GitHub with valid execution histories, only 24% could be re-run successfully, and only 4% reproduced the same results.
  • JetBrains (2020): More than 36% of Jupyter notebooks on GitHub had invalid execution histories — cells run out of order.

marimo is architecturally immune to both findings. There is no mechanism to run cells out of order. Deleting a cell removes its variables from memory. Nature journal highlighted marimo specifically for its reproducibility guarantees for computational scientists.

The Multi-Language Question

Short Answer: marimo Is Python-Only by Design, Not Accident

This is not a gap in the backlog or a feature that wasn't prioritized yet. The marimo team has made this a conscious architectural choice.

GitHub Issue #6620 — "R language support" marimo team collaborator Trevor Manz closed this issue as "not planned" with the following response: "marimo is tightly coupled to Python, which allows us to make marimo extra useful for Python developers." The team's v1.0 roadmap (January 2025) contains no mention of multi-language support. The six roadmap goals are all Python-centric: SQL integration, PEP 723 dependencies, developer experience, sharing/embedding, scalability, and enterprise features.

The Architectural Reason

marimo's reactive execution model depends on a specific technical mechanism that doesn't generalize to other languages without a complete redesign:

  1. Each cell's code is parsed with Python's ast.parse() to produce an Abstract Syntax Tree.
  2. A custom ScopedVisitor traverses the AST to extract global variable definitions (what this cell creates) and references (what this cell reads).
  3. A DirectedGraph uses this information to build the dependency edges between cells.
  4. When a cell changes, only the cells downstream in this graph are re-executed.

This mechanism is entirely Python-specific. There is no ast.parse() for R code, no ScopedVisitor for Julia. To achieve the same reactive DAG for R, you would need to write a full R static analyzer from scratch. To achieve it across multiple languages in one notebook, you would need inter-language variable bridging (passing a Python variable into an R cell, for example) — a fundamentally different and much harder problem.

marimo's approach (Python-first)

Tight coupling to one language enables deep integration: static analysis for reactive DAG, pure-Python file format, scriptable and importable notebooks, PEP 723 dependency pinning. Language depth over language breadth.

Jupyter's approach (language-agnostic)

Abstracting language specifics into the kernel process enables any language to participate. But this abstraction prevents the kind of deep analysis marimo uses for reactivity — Jupyter cannot build a DAG because it never sees inside cells.

SQL: The Exception That Proves the Rule

SQL is the one non-Python language marimo supports, and how it works reveals why other languages don't: SQL cells are syntactic sugar over Python. Under the hood, mo.sql("SELECT * FROM df") is a Python function call. The result is a Python DataFrame that participates in the Python DAG. The SQL parser only needs to identify which Python variable names appear in the SQL query — a much simpler analysis than building a full DAG for R or Julia code.

This approach can't be generalized to R or Julia. An R cell wouldn't just be calling an R function from Python — it would need its own execution environment, its own variable scope, its own static analyzer for reactivity, and a bridge to exchange values with Python cells.

What It Would Take to Add R or Julia to marimo

For completeness, here is what the engineering work would require. This is not speculative — it follows directly from the architecture as documented.

ComponentWhat's NeededDifficulty
Language-specific static analyzers A dependency extractor for R (e.g., using rpy2 + R's parse() function, or a dedicated Rust-based R parser) and for Julia (using Julia's metaprogramming or JuliaCall). Each must extract variable definitions and references in the same format marimo's DAG uses. Hard
Cross-language DAG If a Python variable is used in an R cell, the DAG edge must cross language boundaries. This requires either a common data representation or explicit type conversion at each crossing. Analogous to how rpy2 converts between Python and R types — but integrated into the reactive runtime. Very hard
Multi-runtime dispatch in Kernel The Kernel class currently dispatches all cell execution to CPython via exec(). Multi-language support would require detecting which runtime to use per cell and dispatching to the appropriate interpreter (Rscript, Julia runtime). Hard
UI element binding across languages Currently, mo.ui.slider is a Python object — its .value attribute is read by Python cells. Binding slider values to R or Julia variables would require a foreign-function interface layer on both sides. Very hard
File format changes The current .py format (Python functions decorated with @app.cell) cannot represent R or Julia cells. A new polyglot format would be needed — similar to Quarto's .qmd with {python} / {r} / {julia} blocks. Moderate
Community-proposed workaround GitHub Issue #6620 suggests a pragmatic workaround: keep Python as the primary engine, use rpy2 to call R from within Python cells. R objects are accessed as Python variables via rpy2's conversion layer. This preserves the Python DAG while allowing R library code to run. It's not true "R cells" — R code lives inside Python string arguments — but it satisfies many use cases without requiring architectural changes.

Language-Specific Analogues: Pluto.jl

The reactive notebook concept is not limited to Python. The Julia ecosystem has Pluto.jl — and importantly, Pluto.jl directly inspired marimo's design.

Pluto.jl (Julia)

Pluto.jl is a reactive, DAG-based notebook for Julia developed at MIT. Like marimo, it:

  • Builds a dependency graph from cell code (using Julia's metaprogramming for AST analysis)
  • Automatically re-runs downstream cells when upstream variables change
  • Forbids variable redefinition across cells
  • Stores notebooks as plain .jl files (not JSON)
  • Supports running as interactive web applications
  • Provides built-in package reproducibility (exact package versions embedded in the file)

The marimo team explicitly credits Pluto.jl as a design inspiration: "Our inspiration comes from many places and projects, especially Pluto.jl, ObservableHQ, and Bret Victor's essays."

The key insight from Pluto.jl Pluto.jl achieves the same reactive guarantees as marimo — but for Julia, not Python. It uses Julia's own metaprogramming facilities to parse cell ASTs and build the dependency graph. This demonstrates that the reactive model is language-achievable — but it requires language-native static analysis. There is no cross-language shortcut. The solution for Julia is Pluto.jl. The solution for Python is marimo. A single product that does both doesn't currently exist.

Bottom Line

Choose marimo if you work primarily in Python, value reproducibility and clean version control, want to turn notebooks into deployed apps without extra tooling, or find Jupyter's hidden state problems frustrating. The reactive model requires adjusting some habits but solves real problems structurally.

Choose Jupyter if you work in R, Julia, or another non-Python language; if you need to interoperate with an existing JupyterHub deployment; if you have a large team with established Jupyter workflows; or if you rely on specific Jupyter extensions with no marimo equivalent.

They are not directly substitutable for multi-language workloads. marimo is Python (+ SQL). Jupyter is everything. For pure-Python data science workflows starting fresh in 2026, marimo's architecture is the better foundation — but it requires accepting that R and Julia users on your team will need a different tool.

Migrating from Jupyter to marimo

marimo provides a migration command: marimo convert notebook.ipynb -o notebook.py. Things to watch for:

  • Cells that share a variable name will fail with MultipleDefinitionError. Rename to unique names per cell.
  • IPython magic commands (%timeit, !ls) will fail. Replace with marimo equivalents or subprocess.run().
  • Cells that depend on out-of-order execution must be restructured. The DAG enforces a consistent order.
  • Cell outputs are not preserved in the .py file — re-execute to see results.
