Stateless GitHub Publishing:
Using the Contents API from Ephemeral Environments

Apr 2026 GitHub API CI/CD serverless AI agents
TL;DR: Ephemeral compute — containers, Lambda, serverless functions, AI agents — can't rely on persistent SSH keys, git credentials, or local clones. The GitHub Contents API solves this: pure HTTP calls let you read and write individual files with no git binary, no local state, and no credentials stored on disk. This is a complete walkthrough of how it works, when to use it, and what to watch out for.

1. The Problem

Modern compute is increasingly ephemeral. Container runtimes spin up for seconds, Lambda functions live for milliseconds, CI/CD workers are discarded after every build, and AI coding agents run inside sandboxed sessions with no persistent home directory.

The traditional git workflow assumes statefulness: you clone a repo, maintain SSH keys, manage credentials in ~/.gitconfig, and push from a workspace that persists between sessions. None of that is available in ephemeral environments.

Options you might reach for — and why they fall short:

  • Clone + push: Requires git binary, SSH keys or a credential helper, and a writable filesystem. Containers and Lambda often have none of these configured.
  • Deploy keys: Better, but still require key management and a persistent SSH agent. Leaks in logs are hard to audit.
  • GitHub Actions built-in token: Works, but only inside GitHub Actions — not useful for external agents, IoT devices, or arbitrary cloud functions.

The better answer, for single-file updates, is the GitHub Contents API. No git binary. No persistent disk state. Pure HTTPS with a token.

2. The GitHub Contents API

GitHub exposes a REST endpoint for reading and writing individual files in a repository. It handles all the git internals server-side — you just send the content, and GitHub creates the blob, tree, and commit for you.

Read a file

# GET the current file (also returns the SHA you need to update it)
GET https://api.github.com/repos/owner/repo/contents/path/to/file.html

Authorization: token ghp_xxxxxxxxxxxx
Accept: application/vnd.github+json

Response includes content (base64-encoded file body) and sha (the blob SHA, required for updates).

Write or update a file

# PUT creates the file (if new) or updates it (if SHA provided)
PUT https://api.github.com/repos/owner/repo/contents/path/to/file.html

Authorization: token ghp_xxxxxxxxxxxx
Content-Type: application/json

{
  "message": "your commit message",
  "content": "<base64-encoded file content>",
  "sha":     "<blob SHA from GET, required for updates>"
}

GitHub responds with 200 (updated) or 201 (created), and returns the new commit metadata.

3. How It Works — Step by Step

  1. Generate content in memory. Your function, agent, or container produces whatever it wants to publish — an HTML page, a JSON report, a markdown file — as a string in memory. No disk write required.
  2. Base64-encode it. The API requires content as a base64 string. In Python: base64.b64encode(content.encode()).decode(). In curl: pipe through base64.
  3. GET the file to retrieve its SHA. If the file already exists, you must include its current blob SHA in the PUT request, or the API returns a 422 conflict error. The SHA comes from the GET response's top-level sha field.
  4. PUT with content + SHA. Send the base64 content, commit message, and SHA (omit SHA for new files). GitHub handles the entire git object model server-side.
  5. GitHub creates the commit. The response includes the new commit SHA, author metadata, and URLs. No local git state was modified or required.

4. Code Examples

Python

import urllib.request, base64, json

TOKEN   = "ghp_xxxxxxxxxxxx"
OWNER   = "owner"
REPO    = "repo"
PATH    = "output/report.html"
HEADERS = {
    "Authorization": f"token {TOKEN}",
    "Accept": "application/vnd.github+json",
}

# 1. Generate content
content = "<html><body>Hello from an ephemeral agent!</body></html>"

# 2. GET existing file (to retrieve SHA for update)
url = f"https://api.github.com/repos/{OWNER}/{REPO}/contents/{PATH}"
req = urllib.request.Request(url, headers=HEADERS)
try:
    r = urllib.request.urlopen(req)
    sha = json.loads(r.read()).get("sha")
except urllib.error.HTTPError:
    sha = None

# 3. PUT to create or update
payload = {
    "message": "publish: update report",
    "content": base64.b64encode(content.encode()).decode(),
}
if sha:
    payload["sha"] = sha

body = json.dumps(payload).encode()
put_req = urllib.request.Request(url, data=body, headers={**HEADERS, "Content-Type": "application/json"}, method="PUT")
resp = urllib.request.urlopen(put_req)
result = json.loads(resp.read())
print(resp.status, result["commit"]["sha"])

curl

# Step 1: get current SHA (if file exists)
SHA=$(curl -s   -H "Authorization: token ghp_xxxxxxxxxxxx"   "https://api.github.com/repos/owner/repo/contents/output/report.html"   | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('sha',''))")

# Step 2: base64-encode content
ENCODED=$(base64 -w 0 report.html)

# Step 3: push
curl -s -X PUT   -H "Authorization: token ghp_xxxxxxxxxxxx"   -H "Content-Type: application/json"   -d "{"message":"publish: update report","content":"$ENCODED","sha":"$SHA"}"   "https://api.github.com/repos/owner/repo/contents/output/report.html"
Note
If the file does not yet exist, omit the "sha" field entirely. The API will create the file. If the file exists and you omit SHA, you get a 422 Unprocessable Entity error.

5. Comparison: Git CLI vs Contents API

Dimension Git CLI workflow Contents API
Requires git binary Yes No
Requires persistent disk Yes (local clone) No
Credential storage SSH key or credential helper Bearer token in request header
Multi-file atomic commit Yes No (one file per PUT)
Binary files Full support Base64 overhead; 100 MB file limit
Branching / merging Full git model Single-file only
Works from Lambda / containers Complex setup Works by default
Works from AI agents / IoT Rarely practical Yes
Rate limit N/A (push-based) 5,000 requests/hour (authenticated)

6. When to Use This Pattern

Good fit
  • Single-file publish from CI/CD (reports, status pages, generated HTML)
  • AI agents publishing research pages or documentation
  • Serverless functions that generate and store a file per invocation
  • IoT or embedded devices pushing readings to a GitHub-hosted datastore
  • Environments where installing git is impractical or forbidden
  • Quick file updates in scripts where a full git workflow would be overhead
Poor fit
  • Multi-file atomic commits (no transaction support across files)
  • Large binary files (base64 overhead, 100 MB API limit)
  • Branching, rebasing, or complex merge workflows
  • High-volume writes at scale (rate limits become a concern)
  • Repositories with branch protection requiring signed commits

7. Gotchas

SHA is mandatory for updates

The Contents API uses the blob SHA as an optimistic concurrency lock. If you skip the GET step and just PUT, you will get a 422 Unprocessable Entity the moment the file already exists. Always GET first when you're not certain if the file exists.

No atomic multi-file commits

Each PUT is its own commit. If you need to update five files "atomically," the Contents API is the wrong tool. Use the lower-level Git Trees API instead, which lets you build a tree object and create a single commit pointing to it.

Rate limits

Authenticated requests get 5,000 per hour. Each GET + PUT pair costs 2 requests. For most publishing use cases this is ample; for high-frequency automation, track the X-RateLimit-Remaining response header.

Default branch only (for simplicity)

You can specify a branch in the GET request (?ref=my-branch) and in the PUT body ("branch": "my-branch"). Without specifying, the API reads and writes to the repository's default branch. Confirm what your default branch is if it's not main.

Token scope

The token must have the contents: write permission (fine-grained) or repo scope (classic). Keep tokens scoped as narrowly as possible — a write token for one repo, not all repos.
