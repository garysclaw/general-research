What Is LLM Wiki?

LLM Wiki is a conceptual pattern for personal knowledge management, published by Andrej Karpathy (former Tesla AI Director, OpenAI co-founder, creator of NanoGPT and micrograd) in early April 2026 as a GitHub Gist. It is not a product, not a library, and not a protocol — it is a blueprint for how to structure the relationship between an LLM agent and a personal knowledge base.

The core idea: instead of searching raw documents at query time (as RAG does), use an LLM to compile your sources into a structured, interlinked markdown wiki once, and then keep that wiki maintained and up-to-date as new sources arrive. The LLM acts as a persistent librarian rather than a search-and-answer machine.

"Rather than treating the LLM like a search engine over your documents, treat it like an expert who has read everything and written up their notes — and you can ask those notes questions."

— Karpathy's framing, paraphrased from the gist

Karpathy deliberately published this as an "idea file" rather than production code. The gist is a plain-text prompt/blueprint that you paste into an LLM agent, which customises an implementation for your specific setup. This framing — the "idea file" as a shareable unit of agent instruction — was itself considered noteworthy by the community.

Reception The original gist received 5,000+ stars within days. Karpathy's tweet announcing it got 17,000+ likes and 3 million+ views in 48 hours. Two separate threads reached the Hacker News front page (295 points, 94 comments on the main thread). Dozens of independent implementations appeared on GitHub within the first week.

The Problem It Solves

Standard LLM use has a fundamental statefulness problem:

  • Every new chat session starts with no memory of previous ones
  • RAG systems re-search and re-synthesise raw documents on every query — no accumulated understanding
  • Knowledge stays fragmented across chat histories, PDFs, browser tabs, and notes apps
  • Cross-referencing and synthesis must be redone from scratch every time you ask a related question

Karpathy's analogy: current LLM workflows treat the model like a compiler that discards all intermediate work after each run. A better model is a librarian who reads everything, writes up their notes, keeps those notes organised, and you consult the notes — not the raw source material.

The compiled wiki is the intermediate work that gets saved. Ingesting a new document is a one-time cost. All future queries run against the synthesised knowledge base, not the raw sources.

How It Works — Architecture

The system has three layers:

┌─────────────────────────────────────────────┐ │ RAW SOURCES (read-only) │ │ PDFs · URLs · Papers · YouTube transcripts│ │ raw/ directory — never modified by LLM │ └──────────────────────┬──────────────────────┘ │ ingest ▼ ┌─────────────────────────────────────────────┐ │ THE WIKI (LLM-maintained) │ │ wiki/ — markdown files with [[wikilinks]] │ │ Concept pages · Entity pages · Comparisons │ │ Cross-references · Summaries · Indexes │ └──────────────────────┬──────────────────────┘ │ query ▼ ┌─────────────────────────────────────────────┐ │ QUERIES │ │ Answered from compiled wiki, not raw docs │ │ Valuable answers filed back as new pages │ └─────────────────────────────────────────────┘

A fourth component sits above all three: the schema file (CLAUDE.md or AGENTS.md), which defines the wiki's structure, naming conventions, link formats, and the agent's workflow rules. This is the "idea file" — it is what Karpathy actually shared in the gist.

Directory layout

knowledge-base/
├── CLAUDE.md           ← schema / agent instructions (the "idea file")
├── raw/                ← immutable source documents (never modified)
│   ├── paper-2024.pdf
│   ├── article-url.md
│   └── yt-transcript.txt
└── wiki/               ← LLM-compiled knowledge (always evolving)
    ├── INDEX.md
    ├── concepts/
    │   ├── transformer-architecture.md
    │   └── attention-mechanism.md
    └── entities/
        ├── andrej-karpathy.md
        └── openai.md

Scale target from Karpathy's own setup: approximately 100 articles and 400,000 words. The approach works reliably up to roughly 50,000–100,000 tokens (~100–200 wiki pages) within a single agent context.

The Three Core Operations

1. Ingest

When a new source is added to raw/, the LLM:

  1. Reads the source document in full
  2. Identifies key concepts, entities, facts, and relationships
  3. Integrates the new information into existing wiki pages (extending, correcting, adding cross-references)
  4. Creates new pages where concepts don't yet exist
  5. Updates the index and adds [[wikilinks]] between related pages
  6. Logs the ingest in an activity file

Knowledge is compiled once at ingest time. Subsequent queries don't re-read the source.

2. Query

Questions are answered from the compiled wiki — not from raw sources. Because the wiki already contains synthesised, cross-referenced knowledge, answers are faster and more coherent than querying raw documents. Valuable query responses that represent new synthesis can be filed back into the wiki as new pages.

3. Lint

Periodic health-check operation. The LLM reviews the wiki for:

  • Contradictions between pages (newer source contradicts older synthesis)
  • Orphaned pages (no inbound [[wikilinks]])
  • Stale claims (information that has likely been superseded)
  • Dead links and broken references
  • Knowledge gaps (concepts mentioned but not yet given their own page)

Implementations in the Wild

There is no single canonical llm-wiki repository. The ecosystem is fragmented — dozens of independent implementations appeared on GitHub within the first week. Notable ones:

lucasastorian/llmwiki Most Stars (~148)
Full-stack web app (Next.js + FastAPI + Supabase + MCP). Hosted version at llmwiki.app. The most polished implementation but the furthest from the original file-system-first concept.
iamsashank09/llm-wiki-kit
Python toolkit with PDF/URL/YouTube ingest, SQLite FTS5 full-text search, MCP server support, graph visualization. Closest to the original pattern in spirit — stays file-system oriented.
ekadetov/llm-wiki
Claude Code plugin for Obsidian. Python + Shell. Designed to run the ingest/lint operations directly from within Claude Code against an Obsidian vault.
MehmetGoekce/llm-wiki
L1/L2 cache architecture — hot/warm/cold tiers for wiki pages. Supports both Logseq and Obsidian as the viewer layer.
ussumant/llm-wiki-compiler
Claude Code plugin that compiles markdown documents into topic-based wiki pages. Explicitly designed as a Claude Code skill.
LLM Wiki v2 (gist) Extension
Community extension of the original pattern. Adds: confidence scoring per claim, memory lifecycle management (active/archived/deprecated), typed knowledge graph relationships, multi-agent sync. More rigorous but more complex.
CacheZero (Show HN)
Packaged the entire concept as a single npm install. Appeared on Hacker News as a Show HN post. NPM-first approach — lowest friction for JS developers.

LLM Wiki vs. RAG

RAG (Retrieval-Augmented Generation) and LLM Wiki solve adjacent but different problems. The comparison that dominated the Hacker News discussion:

DimensionLLM WikiRAG
When knowledge is processedAt ingest time — once, compiled into the wikiAt query time — re-retrieved and re-synthesised every query
Infrastructure requiredZero — plain markdown files on diskVector database + embedding pipeline + retrieval service
Scale ceiling~50k–100k tokens (~100–200 wiki pages)Millions of documents with appropriate indexing
Cross-referencesPersistent, maintained wikilinksAd hoc, rediscovered per query
FreshnessRequires manual re-ingest + lint cycleNear-real-time via pipeline re-indexing
Multi-userWrite conflicts — difficultNative, by design
Human readabilityFully readable, auditable markdownMachine-oriented vector embeddings
Token efficiencyUp to 95% reduction vs. loading raw docsHigher per-query cost depending on chunk size
Best forPersonal, stable, mid-sized knowledge domainsLarge, dynamic, multi-user document corpora
The recommended approach: hybrid LLM Wiki and RAG are not mutually exclusive. One practical hybrid: stable, core knowledge (architecture decisions, key concepts, personal notes) maintained as an LLM Wiki loaded into the system prompt; RAG used for dynamic or large-scale content (recent papers, large codebases, customer call transcripts). Each does what it's better at.

LLM Wiki vs. Traditional PKM Tools

ToolKey Difference
Obsidian / Logseq / Roam The human does the synthesis, cross-referencing, and maintenance. LLM Wiki offloads all of that to the agent. Obsidian is actually recommended as the viewer for an LLM Wiki — the tools are complementary, not competing.
Notion / Confluence Static wikis go stale because humans don't maintain them. The LLM Wiki pattern delegates maintenance to the agent, keeping it current. The "company bible" use case (LLM maintains an internal wiki from Slack threads, meeting transcripts, and project docs) was widely discussed on HN.
Knowledge Graphs (Neo4j) Formally verified, queryable graph relationships — but requires Cypher/SPARQL expertise and significant engineering. LLM Wiki trades formal graph semantics for zero infrastructure and human readability. LLM Wiki v2 adds typed relationships to partially close this gap.
Zettelkasten Philosophically similar — atomic notes, permanent links. The key difference is authorship: Zettelkasten is human-authored (learning through writing), LLM Wiki is agent-authored (you delegate the writing). This difference was a major point of debate on HN.

Criticisms and Limitations

Technical limitations

  • Scale ceiling is real — works reliably at ~100–200 pages. Above that, context window limits, slow lint cycles, and cross-reference management become problematic. Not an enterprise solution.
  • Multi-user write conflicts — designed for single-person use. Two agents writing to the same wiki simultaneously will produce conflicts. No built-in resolution mechanism.
  • No real-time freshness — re-ingesting updated sources and running lint is a manual, periodic operation. For rapidly changing domains, this is a significant disadvantage vs. RAG.
  • Context window obsolescence — as models gain larger context windows, the need for a compiled intermediate layer may diminish. If you can load 200 raw documents in context, why compile them?

Hallucination accumulation

The compounding error problem Errors synthesised into wiki pages can propagate into subsequent ingest cycles — an LLM reads a page with an incorrect claim and treats it as authoritative when integrating new sources. Over many ingest cycles, errors can become entrenched and hard to audit back to their origin. The lint operation is intended to catch contradictions, but it can't catch plausible-but-wrong synthesis.

Philosophical criticisms (from HN)

  • "Not genuinely novel" — several developers noted that coding assistants and memory-augmented agents had been doing this for over a year before Karpathy's post. The novelty is in the naming and framing, not the technical concept.
  • Cognitive value loss — organising and synthesising information yourself produces insight ("shower thoughts"). The act of writing forces understanding. Automating this removes a learning step. Several HN commenters argued the value of a Zettelkasten is in the building, not the built thing.
  • Maintenance burden shifts, not disappears — you still need to curate what goes into raw/, review what the LLM synthesises, and run lint cycles. The automation reduces but doesn't eliminate human judgment.

Use Cases

Developer context persistenceExcellentAI coding agents that remember project architecture, decisions, and conventions across sessions. ekadetov/llm-wiki and ussumant/llm-wiki-compiler target this directly.

Large-scale enterprise knowledge managementPoorScale ceiling, no access control, write conflicts, audit challenges. RAG or purpose-built enterprise tools are better here.
