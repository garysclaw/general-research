Docker Containers vs Sandboxes

A deep dive into isolation technologies: Docker, gVisor, Firecracker, Kata Containers, WASM, and OS-level policy sandboxes — understanding the isolation spectrum, the real security implications of shared kernels, and when to reach for each tool.

The Core Confusion: "Sandbox" Is an Overloaded Term

Key Mental Model

Containers are a form of sandboxing — they isolate processes using Linux namespaces and cgroups. But when engineers say "sandbox," they often mean something with deeper isolation guarantees than a container provides. The question is never sandbox vs container; it is always which point on the isolation spectrum do you need?

"Sandbox" means different things in different contexts. A browser sandbox (V8 Isolate) is vastly different from a gVisor sandbox, which is vastly different from a Firecracker microVM. What they share is the intent: constrain what untrusted code can do. The mechanisms — and therefore the security guarantees — differ enormously.

Isolation exists on a spectrum. Moving right means stronger isolation and higher overhead:

Every step to the right reduces the attack surface but increases startup latency and resource overhead. The right choice depends on who is running the code, what the blast radius of an escape is, and what your performance budget allows.

What Is a Container? (Docker, OCI)

A container is not a virtual machine. It is a collection of Linux kernel features that create the illusion of an isolated environment while sharing the host's kernel. Docker and the OCI (Open Container Initiative) runtime spec standardise how these features are assembled.

Linux Namespaces — the Isolation Primitive

Namespaces partition kernel resources so that one set of processes sees one set of resources and another set of processes sees a different set. Docker uses six namespaces:

  • pid — process tree isolation; container PID 1 appears as init inside, but has a different PID on the host
  • net — separate network stack: interfaces, routes, iptables rules, sockets
  • mnt — private mount table; the container's filesystem view
  • uts — independent hostname and domain name (UTS = UNIX Time-sharing System)
  • ipc — isolated System V IPC and POSIX message queues
  • user — UID/GID remapping; UID 0 inside maps to unprivileged UID outside (when enabled)
Critical Point

Namespaces do not prevent syscalls. A process inside a container can still call open(2), mmap(2), clone(2), or any other syscall. Namespaces only change what the kernel returns — not whether the syscall is allowed. This is the fundamental distinction from a gVisor or VM sandbox.

cgroups — Resource Governance

Control Groups (cgroups v2) enforce resource limits. Docker translates --memory, --cpus, and --blkio-weight flags into cgroup entries under /sys/fs/cgroup/. A container hitting its memory limit gets OOM-killed by the kernel; it cannot steal memory from other containers. However, cgroups are about fairness and limits, not security isolation — they don't prevent a container from making privilege-escalation syscalls.

Union Filesystems (overlay2)

Docker images are built from immutable layers stacked with OverlayFS. Each image layer is a directory diff. At runtime, a thin writable layer sits on top (copy-on-write). This makes images share common base layers on disk and allows containers to start without copying gigabytes of data. The container sees a merged view but writes only to its own layer.

The Key Security Implication: Shared Host Kernel

The Single Biggest Security Fact About Containers

Every Docker container on a host shares the same Linux kernel. If a container exploits a kernel vulnerability — a buffer overflow in a syscall handler, a race condition in io_uring, a privilege escalation via a kernel data structure — it can escape to the host and own every other container on the machine. There is no kernel-level boundary between containers; only namespace and cgroup accounting enforced by that kernel.

Performance Characteristics

  • Startup time: ~50–150ms for a typical container (image already pulled)
  • CPU overhead: <1% vs bare metal for most workloads (namespace lookups are negligible)
  • Memory overhead: ~1–5 MB per container daemon bookkeeping; no hypervisor tax
  • I/O: overlay2 adds one extra copy-on-write on first write per file; subsequent writes are native speed

The Sandbox Landscape — Five Categories

A. Kernel-Intercepting Sandboxes — gVisor (runsc)

Google Open Source Recommended for untrusted code

gVisor is Google's answer to the shared-kernel problem. Instead of letting container processes talk directly to the host kernel, gVisor inserts a userspace Go binary called the Sentry that intercepts every Linux syscall the container makes.

The Sentry is a Linux kernel implemented in Go — it understands open(2), read(2), fork(2), etc., and re-implements them in userspace. The container process thinks it's talking to a real Linux kernel. In reality, the Sentry handles the call and only makes a tiny set of carefully audited syscalls down to the actual host kernel. This dramatically shrinks the host kernel's attack surface from hundreds of syscalls to a handful.

gVisor runs in two modes:

  • ptrace mode — uses PTRACE_SYSCALL to intercept calls. Works on any Linux machine. Slower (~2–5x overhead for syscall-heavy workloads).
  • KVM mode — the Sentry runs inside a KVM virtual machine, with hardware assistance for syscall interception. Faster (~20–30% overhead), requires KVM device.
Real-World Deployment

Google Cloud Run and GKE Sandbox use gVisor as the default runtime for all user workloads. Every Cloud Run container you deploy is isolated by gVisor's Sentry, not bare runc. This is how Google safely multi-tenants millions of customer containers on shared hosts.

Trade-offs:

  • ~20–30% CPU overhead (KVM mode) to ~2–5x overhead (ptrace mode) for syscall-intensive code
  • Some syscalls are not yet implemented — applications using io_uring, obscure ioctls, or newer kernel features may fail
  • Compatibility testing required before production adoption

B. VM-Based Containers — Kata Containers / Firecracker

AWS OpenInfra Foundation Strongest container-compat isolation

Rather than intercepting syscalls, Kata Containers give each container (or pod) its own lightweight virtual machine with its own kernel. The container runtime (containerd or CRI-O) launches a minimal VM using QEMU, AWS Firecracker, or Cloud Hypervisor, boots a stripped-down Linux kernel inside it, and then starts the container inside that VM.

Firecracker (developed by AWS for Lambda and Fargate) is purpose-built for this use case. It is a minimal VMM (Virtual Machine Monitor) written in Rust that exposes only a small set of emulated devices (virtio-net, virtio-block, serial). This tiny surface area means:

  • Boot time: ~125ms for a Firecracker microVM (vs ~1–2s for QEMU)
  • Memory overhead: <5 MB per microVM for the VMM itself
  • True kernel isolation: a CVE in the host kernel does not affect code running inside the VM kernel
Who Uses This

AWS Lambda runs every function invocation inside a Firecracker microVM. AWS Fargate uses Firecracker for ECS/EKS workloads. Fly.io runs customer VMs on Firecracker. This is the technology enabling true multi-tenant serverless compute where customer A's kernel panic cannot affect customer B.

Trade-offs:

  • ~125ms cold start vs ~50ms for containers (still fast enough for serverless)
  • Higher baseline memory than containers (one kernel + VMM per workload)
  • Requires KVM hardware support — no nested virtualisation in most cloud instances unless explicitly enabled

C. OS-Level Policy Sandboxes — seccomp, AppArmor, SELinux

Built into Linux Works inside containers

These are not separate runtimes — they are Linux kernel security mechanisms that restrict what a process (including a container process) is allowed to do. Docker applies them by default, and they can be tightened significantly.

seccomp (Secure Computing Mode) — a kernel feature that installs a BPF filter on the syscall path. Every syscall is checked against the filter before execution. Docker's default seccomp profile blocks ~44 syscalls including ptrace, kexec_load, perf_event_open, bpf, and others that have historically been used in container escapes. You can tighten this to an allowlist of only the syscalls your application actually needs.

AppArmor — a Linux Security Module that enforces path-based mandatory access control. Docker's default AppArmor profile (docker-default) restricts mount operations, denies access to /proc/sysrq-trigger, and limits certain capabilities. Custom profiles can lock down file paths, network access, and capability use.

SELinux — uses security labels (contexts) on files, processes, and sockets. Mandatory access control enforced by the kernel's security framework. Common in RHEL/Fedora environments. Container runtimes assign an svirt_lxc_net_t type to container processes, constraining what they can access on the host.

Important Nuance

seccomp, AppArmor, and SELinux work inside the shared-kernel model. They reduce the syscall surface available to a container, but they do not eliminate the host kernel exposure entirely. A zero-day in a syscall that is on the allowlist can still result in a host escape. They are defence-in-depth, not isolation boundaries.

D. Language/Runtime Sandboxes — WASM, V8 Isolates, Deno

Cloudflare Workers Fastly Compute ~0ms startup

These sandboxes operate at the language/runtime level rather than the OS level. They do not use namespaces or VMs — they sandbox at the instruction level.

WebAssembly (WASM) — a binary instruction format that runs in a capability-based sandbox. WASM programs cannot make syscalls at all; the runtime (Wasmtime, WasmEdge, browser engine) mediates all I/O through explicit imports. WASI (WebAssembly System Interface) defines a capability model: you must explicitly grant access to directories, sockets, and clocks. Deny-by-default, with explicitly granted permissions.

V8 Isolates (used by Cloudflare Workers) — V8 runs JavaScript inside isolated heaps called "Isolates". Each Worker gets its own Isolate with no shared memory. A single OS process can host thousands of Isolates simultaneously. Startup time is ~0ms for warm Isolates, ~1–5ms for cold. The trade-off: JavaScript only (or languages compiled to JS/WASM), and the V8 security model assumes bugs in V8 itself are fixed promptly.

Deno — a JavaScript/TypeScript runtime built on V8 with an explicit permission system. Network access, file system access, and environment variable access are all denied by default and must be granted with flags like --allow-net=api.example.com and --allow-read=/tmp. This makes the permission model auditable at launch time.

Trade-offs: language-specific (or requires compiling to WASM/JS), not general-purpose, cannot run arbitrary Linux binaries.

E. Hypervisor / Full VM — QEMU, VirtualBox, KVM

Highest overhead Strongest isolation

Traditional virtual machines emulate complete hardware — CPU, memory, disk controllers, network cards. Each VM boots its own kernel and userspace OS from scratch. The hypervisor (QEMU/KVM, VirtualBox, VMware ESXi, Hyper-V) manages resource allocation and hardware emulation. The attack surface from a guest to the host is the hypervisor itself — a much smaller and more audited codebase than an entire Linux kernel.

  • Startup: 5–30 seconds (full OS boot)
  • Memory: 256 MB–several GB per VM (full OS stack)
  • Isolation: complete kernel + userspace separation
  • Use case: different OS (Windows on Linux host), maximum compliance requirements, legacy software

When to Use What — Decision Matrix

Use Plain Docker When

  • Running internal services you trust — your own application code, written by your team
  • Development environments — fast iteration matters more than maximum isolation
  • CI/CD build runners where the code being built is from your own repositories
  • Performance is critical and you cannot accept even a 20% CPU overhead
  • Multi-tenant where all tenants are your own team (not external customers)

Use Docker + Strict seccomp/AppArmor When

  • Production workloads — defence in depth costs almost nothing in performance
  • You want 95% of Docker convenience with a meaningfully better security posture
  • Network-exposed services — reduces the syscall surface an exploited app can leverage
  • You need to demonstrate security controls for audit purposes without changing your runtime

Use gVisor (runsc) When

  • Running untrusted or third-party code — npm packages, user-submitted Dockerfiles
  • Multi-tenant SaaS where one customer's container could attack another
  • You need Linux ABI compatibility (gVisor implements the Linux syscall interface) but stronger isolation than containers
  • You are already on Google Cloud Run or GKE — gVisor is the default there
  • Compatibility with your workload has been validated (test first — some apps break)

Use Kata Containers / Firecracker When

  • Running truly untrusted code — user uploads, arbitrary code execution services
  • Building a serverless functions platform (AWS Lambda model)
  • A kernel vulnerability would be catastrophic — financial systems, health data
  • Compliance requirements (PCI DSS, HIPAA, FedRAMP) explicitly require VM-level isolation between workloads
  • You can accept ~125–300ms cold start overhead

Use WASM When

  • Building a plugin system or extension point — WASM is the new DLL
  • Edge compute (Cloudflare Workers, Fastly Compute) where ~0ms startup and small binary size matter
  • Language-agnostic sandboxing — compile Rust, Go, C, Python, etc. to WASM
  • You need deterministic, reproducible execution (WASM is well-specified)
  • The workload is computationally intensive but does not need arbitrary OS access

Use Plain VMs When

  • Legacy software that cannot be containerised (requires specific kernel version, specific drivers)
  • Different OS entirely — running Windows on a Linux host
  • Maximum isolation with full OS auditing, snapshotting, and rollback
  • The 30-second boot time is acceptable (batch jobs, long-running workloads)

Example 1 — Docker Baseline: Inspect Isolation

# Run a container and inspect its isolation
docker run --rm -it ubuntu:24.04 bash

# Inside the container — check your namespaces
ls -la /proc/self/ns/
# You'll see: cgroup, ipc, mnt, net, pid, pid_for_children, time, uts, user

# Check if memory limit is set (requires cgroups v2)
cat /sys/fs/cgroup/memory.max
# "max" means unlimited; set a limit with: docker run --memory=512m ...

# Verify you can't see host processes
ps aux
# Only container processes visible — pid namespace at work

# Try to mount — should be denied (CAP_SYS_ADMIN not granted by default)
mount -t proc proc /tmp/proc
# mount: /tmp/proc: permission denied.

# Check capabilities (should be a reduced set)
cat /proc/self/status | grep CapEff
# Decode: capsh --decode=<hex>

Example 3 — Maximally Hardened Docker Container

# Drop ALL Linux capabilities, add back only what nginx needs
docker run --rm -it \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --read-only \
  --tmpfs /tmp:size=50m,mode=1777 \
  --tmpfs /var/cache/nginx:size=50m \
  --tmpfs /var/run:size=10m \
  --no-new-privileges \
  --security-opt no-new-privileges:true \
  --security-opt apparmor=docker-default \
  --user 101:101 \
  nginx:alpine

# What each flag does:
# --cap-drop=ALL            Remove all 38 Linux capabilities
# --cap-add=NET_BIND_SERVICE Allow binding to ports <1024 (needed for :80)
# --read-only               Root filesystem is immutable
# --tmpfs ...               Writable in-memory mounts for runtime files
# --no-new-privileges       Prevent setuid binaries from gaining capabilities
# --user 101:101            Run as non-root (nginx user in alpine image)

Example 4 — Install and Run gVisor (runsc)

# ── Install gVisor on Ubuntu/Debian ──
(
  set -e
  ARCH=$(uname -m)
  URL=https://storage.googleapis.com/gvisor/releases/release/latest/${ARCH}
  wget ${URL}/runsc ${URL}/runsc.sha512
  sha512sum -c runsc.sha512
  rm -f runsc.sha512
  chmod a+rx runsc
  sudo mv runsc /usr/local/bin
)

# ── Register runsc as a Docker runtime ──
sudo tee /etc/docker/daemon.json << 'EOF'
{
  "runtimes": {
    "runsc": {
      "path": "/usr/local/bin/runsc"
    }
  }
}
EOF
sudo systemctl reload docker

# ── Run a container with gVisor ──
docker run --runtime=runsc --rm -it ubuntu:24.04 bash

# ── Verify gVisor is intercepting syscalls ──
docker run --runtime=runsc --rm ubuntu:24.04 \
  sh -c "cat /proc/version"
# Output: Linux version 4.4.0 #1 SMP Sun Jan 10 15:06:54 PST 2016 (GVisor)
# Key: "GVisor" in the kernel version string — you're talking to the Sentry, not the real kernel

# ── Compare: same command with regular runc ──
docker run --runtime=runc --rm ubuntu:24.04 \
  sh -c "cat /proc/version"
# Output: Linux version 6.8.0-51-generic (buildd@...) ...
# This is the real host kernel version

# ── gVisor syscall interception demo ──
# Try kexec_load (blocked even if not in seccomp profile, since Sentry never forwards it)
docker run --runtime=runsc --rm ubuntu:24.04 \
  sh -c "python3 -c \"import ctypes; ctypes.CDLL(None).syscall(246, 0, 0, 0, 0)\""
# EPERM — gVisor's Sentry returns permission denied; host kernel never sees it

Example 5 — Firecracker microVM Setup

# ── Check KVM availability ──
ls /dev/kvm && echo "KVM available" || echo "KVM not available"
# Firecracker requires /dev/kvm — check cloud provider docs for nested virt support

# ── Install Firecracker ──
ARCH="$(uname -m)"
release_url="https://github.com/firecracker-microvm/firecracker/releases"
latest=$(basename $(curl -fsSLI -o /dev/null -w %{url_effective} ${release_url}/latest))
curl -L ${release_url}/download/${latest}/firecracker-${latest}-${ARCH}.tgz \
  | tar -xz
sudo mv release-${latest}-$(uname -m)/firecracker-${latest}-${ARCH} /usr/local/bin/firecracker
sudo chmod +x /usr/local/bin/firecracker
firecracker --version

# ── Download quickstart demo images (x86_64) ──
curl -fsSL -o hello-vmlinux.bin \
  https://s3.amazonaws.com/spec.ccfc.min/img/quickstart_guide/x86_64/kernels/vmlinux.bin
curl -fsSL -o hello-rootfs.ext4 \
  https://s3.amazonaws.com/spec.ccfc.min/img/quickstart_guide/x86_64/rootfs/bionic.rootfs.ext4

# ── Start Firecracker API socket ──
rm -f /tmp/firecracker.socket
firecracker --api-sock /tmp/firecracker.socket &

# ── Configure the microVM via REST API ──
# Set kernel
curl -s -X PUT \
  --unix-socket /tmp/firecracker.socket \
  http://localhost/boot-source \
  -H "Content-Type: application/json" \
  -d '{
    "kernel_image_path": "./hello-vmlinux.bin",
    "boot_args": "console=ttyS0 reboot=k panic=1 pci=off"
  }'

# Set rootfs
curl -s -X PUT \
  --unix-socket /tmp/firecracker.socket \
  http://localhost/drives/rootfs \
  -H "Content-Type: application/json" \
  -d '{
    "drive_id": "rootfs",
    "path_on_host": "./hello-rootfs.ext4",
    "is_root_device": true,
    "is_read_only": false
  }'

# Set machine config (1 vCPU, 256 MB RAM)
curl -s -X PUT \
  --unix-socket /tmp/firecracker.socket \
  http://localhost/machine-config \
  -H "Content-Type: application/json" \
  -d '{"vcpu_count": 1, "mem_size_mib": 256}'

# Boot it
curl -s -X PUT \
  --unix-socket /tmp/firecracker.socket \
  http://localhost/actions \
  -H "Content-Type: application/json" \
  -d '{"action_type": "InstanceStart"}'
# MicroVM boots in ~125ms — serial console output appears on stdout

Example 6 — WASM Sandbox with Wasmtime

# ── Install wasmtime ──
curl https://wasmtime.dev/install.sh -sSf | bash
source ~/.bashrc   # or restart shell
wasmtime --version

# ── Write a test program ──
cat > sandbox_test.rs << 'EOF'
use std::fs;
use std::io::Read;

fn main() {
    println!("Hello from WASM sandbox!");

    // Attempt to read a file — only works if --dir is granted
    match fs::read_to_string("/etc/hostname") {
        Ok(content) => println!("Host: {}", content.trim()),
        Err(e) => println!("File access denied: {} (expected in sandbox)", e),
    }
}
EOF

# ── Compile to WASM (requires Rust + wasm32-wasip1 target) ──
rustup target add wasm32-wasip1
rustc --target wasm32-wasip1 sandbox_test.rs -o sandbox_test.wasm

# ── Run WITHOUT any filesystem access (default sandbox) ──
wasmtime sandbox_test.wasm
# Output:
# Hello from WASM sandbox!
# File access denied: failed to find a pre-opened file descriptor (expected in sandbox)

# ── Run WITH /etc granted (explicit capability) ──
wasmtime --dir /etc sandbox_test.wasm
# Output:
# Hello from WASM sandbox!
# Host: my-server-hostname

# ── No network by default either ──
cat > net_test.wat << 'EOF'
(module
  (import "wasi_snapshot_preview1" "sock_accept" (func $sock_accept (param i32 i32 i32) (result i32)))
  (func (export "_start"))
)
EOF
wasmtime net_test.wat
# Missing import or capability denied — network is gated behind explicit WASI permissions

Example 7 — Python Process Sandbox with Resource Limits

#!/usr/bin/env python3
"""Simple process-level sandbox using resource limits + Linux namespaces."""
import subprocess
import tempfile
import os

def run_sandboxed(code: str, timeout: int = 5) -> dict:
    """
    Run untrusted Python code in a subprocess with:
    - Memory limit: 64 MB (RLIMIT_AS)
    - CPU time limit: 2 seconds (RLIMIT_CPU)
    - Max open files: 10 (RLIMIT_NOFILE)
    - Network namespace isolation (unshare --net)
    - PID namespace isolation (unshare --pid --fork)
    - Mount namespace isolation (unshare --mount)
    """
    with tempfile.NamedTemporaryFile(mode='w', suffix='.py', delete=False) as f:
        f.write(code)
        script = f.name

    try:
        result = subprocess.run(
            [
                # Linux namespace isolation — requires CAP_SYS_ADMIN or user namespaces
                "unshare", "--net", "--pid", "--mount", "--fork",
                "python3", "-c",
                f"""
import resource, sys

# Memory: 64 MB — process killed if it tries to allocate more
resource.setrlimit(resource.RLIMIT_AS, (64 * 1024 * 1024, 64 * 1024 * 1024))

# CPU: 2 seconds — SIGXCPU sent when exceeded
resource.setrlimit(resource.RLIMIT_CPU, (2, 2))

# Max open file descriptors: 10
resource.setrlimit(resource.RLIMIT_NOFILE, (10, 10))

# Max child processes: 0 (no fork bombing)
resource.setrlimit(resource.RLIMIT_NPROC, (0, 0))

# Max file size that can be written: 1 MB
resource.setrlimit(resource.RLIMIT_FSIZE, (1024 * 1024, 1024 * 1024))

exec(open({repr(script)}).read())
"""
            ],
            capture_output=True,
            text=True,
            timeout=timeout,
        )
        return {
            "stdout": result.stdout,
            "stderr": result.stderr,
            "returncode": result.returncode
        }
    except subprocess.TimeoutExpired:
        return {"error": "Timeout exceeded", "stdout": "", "stderr": ""}
    except PermissionError as e:
        return {"error": f"unshare failed (need user namespaces): {e}", "stdout": "", "stderr": ""}
    finally:
        os.unlink(script)


# ── Test cases ──
if __name__ == "__main__":
    print("=== Normal code ===")
    print(run_sandboxed('print("Hello from sandbox!")'))

    print("\n=== Network access (blocked — unshare --net creates lo-only namespace) ===")
    print(run_sandboxed('import urllib.request; urllib.request.urlopen("http://example.com")'))

    print("\n=== Fork bomb (RLIMIT_NPROC = 0 prevents any child processes) ===")
    print(run_sandboxed('import os\nwhile True: os.fork()'))

    print("\n=== CPU exhaustion (killed after 2 CPU seconds) ===")
    print(run_sandboxed('while True: pass', timeout=4))

    print("\n=== Memory exhaustion (fails at 64 MB) ===")
    print(run_sandboxed('x = " " * (100 * 1024 * 1024)'))
Security Warning

This Python sandbox approach is educational. For production use of untrusted code execution, use Kata Containers or Firecracker. unshare requires either root or user namespaces enabled (kernel.unprivileged_userns_clone=1), and resource limits can be bypassed by sufficiently motivated attackers with kernel exploits. This is a defence-in-depth layer, not a complete isolation boundary.

        runtime_type = "io.containerd.kata.v2"

Container Escape Techniques — Why Isolation Level Matters

Understanding real escape techniques illustrates concretely why the isolation level you choose has security consequences. Each of these was a real-world vulnerability; some are patched, some are configuration anti-patterns that still exist in production today.

CVE-2019-5736 — runc Binary Overwrite

A malicious container could overwrite the host's runc binary by exploiting a race condition in how runc opens /proc/self/exe. On the next docker exec, the attacker's code ran as root on the host. Affected every runc-based container runtime (Docker, containerd, CRI-O, Podman at the time).

Not exploitable with gVisor (Sentry implements /proc/self/exe in userspace) or Kata/Firecracker (host runc is never invoked inside the guest VM).

CVE-2022-0847 — Dirty Pipe (Linux Kernel)

A kernel bug in pipe buffer handling allowed unprivileged users to overwrite arbitrary read-only file-backed page cache entries — including /etc/passwd and SUID binaries. Container processes could exploit this to write to host files, escaping into the host filesystem with root privileges. Affected Linux 5.8–5.16.11.

Kata Containers and Firecracker were completely immune — the vulnerable host kernel was never directly exposed to container processes. Plain Docker containers on affected kernel versions were exploitable regardless of seccomp profile.

Configuration: --privileged Flag

Running docker run --privileged grants the container all 38 Linux capabilities, full access to every device on the host, and the ability to remount the root filesystem. This is essentially no isolation — a process inside can trivially mount the host filesystem and chroot into it: 

mount /dev/sda1 /mnt/host && chroot /mnt/host
# Now you have a root shell on the host OS

Never use --privileged in production. If you think you need it, you almost certainly need a specific capability instead.

Configuration: Mounted Docker Socket

Mounting the Docker daemon socket into a container: docker run -v /var/run/docker.sock:/var/run/docker.sock gives that container full control over the Docker daemon — which runs as root on the host. Any process inside can run: 

docker run -v /:/host --rm -it ubuntu chroot /host
# Instant root shell on the host — complete escape

This is a trivial escape and is extremely common in CI/CD pipelines that mount the socket for "Docker-in-Docker." Use rootless Docker, Buildah, or Kaniko instead.

The Pattern

Notice that CVE-2019-5736 is a container runtime bug, Dirty Pipe is a kernel bug, and the last two are configuration mistakes. gVisor helps with runtime bugs (the Sentry is the runtime, not runc). Kata/Firecracker helps with both runtime bugs and kernel bugs (different kernel per VM). Neither helps with configuration mistakes — those require operational discipline.

Quick-Reference Decision Table

Scenario Recommended Why
Development environment Docker default Speed and convenience; trust level is your own code
Trusted internal microservices Docker + seccomp Defence in depth at near-zero overhead; reduces blast radius of app-layer exploits
npm install / build pipelines Docker + gVisor Typosquatted packages execute arbitrary code; gVisor limits kernel exposure
User-submitted code execution Kata / Firecracker Full VM isolation; kernel CVEs don't cross VM boundary
Serverless functions (scale to zero) Firecracker / WASM 125ms cold start (Firecracker) or ~1ms (WASM) with strong isolation
Browser / app plugin system WASM / V8 Isolates ~0ms init, capability-based, language-sandboxed, no OS exposure
Compliance (PCI DSS / HIPAA) Kata + AppArmor Auditable VM-level separation; satisfies requirements for workload isolation
Legacy Windows app on Linux host Full VM (QEMU/KVM) Different OS kernel required; containers cannot run a Windows kernel
Edge compute (CDN nodes) V8 Isolates / WASM Thousands of tenants per process; ~0ms tenant switch; no cold start
CI runner accepting PRs from forks gVisor or Kata Fork PRs contain attacker-controlled code; shared-kernel containers are insufficient

Summary

Containers are excellent tools for packaging and running trusted software. Their shared-kernel architecture gives them near-native performance and sub-second startup, but it means every container on a host is one kernel vulnerability away from a host escape. For the majority of internal workloads where you trust the code, this is an acceptable trade-off.

The sandbox landscape provides a range of options for when that trade-off is not acceptable. gVisor adds a userspace kernel interception layer that dramatically reduces the attack surface with modest overhead. Kata Containers and Firecracker provide true VM-level isolation with startup times that are competitive with containers for most workloads. WASM provides capability-based isolation with essentially zero overhead for language-compatible workloads.

The Rule of Thumb

The less you trust the code running in your containers, the further right on the isolation spectrum you should be. Internal services: Docker with seccomp. Third-party packages or build pipelines: gVisor. User-submitted or arbitrary code execution: Kata or Firecracker. No rule applies universally — measure the blast radius of an escape, then choose the isolation boundary that makes that blast radius acceptable.
