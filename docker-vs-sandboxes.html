Docker Sandboxes — AI Agent Isolation Deep Dive

Docker's new Sandbox product is not a container hardening feature — it's a purpose-built microVM system for running AI coding agents like Claude Code, Codex, and Gemini CLI without letting them touch your host machine. Here's how it works, how it compares to plain containers, and how to use it.

Note — this document replaces an earlier version. The original document compared generic container isolation technologies (gVisor, seccomp, Firecracker) without knowing Docker had shipped a specific product called Docker Sandboxes in late 2025. The new version covers the actual product. General container hardening is still covered in the comparison section for context.

What is Docker Sandboxes?

Docker Sandboxes is a purpose-built product for one specific problem: you want to run an AI coding agent against your codebase, but you don't want it to be able to touch anything outside that project directory. Not your SSH keys. Not your AWS credentials. Not your other Docker containers. Not your host filesystem.

It is not a way to run containers more securely in general. It is not a replacement for Kubernetes security policies or Docker Compose hardening. It's specifically an agent execution environment.

The core problem it solves: AI agents need to run arbitrary code, install packages, build Docker images, and spin up containers to test their work. Every existing approach was broken: mounting the Docker socket exposes the full host daemon; Docker-in-Docker requires privileged mode which also escapes isolation; regular containers share the host kernel so a kernel CVE is a host escape. Docker Sandboxes solves all three by giving each agent its own private VM with its own Docker daemon.

Docker's own claim: "Docker Sandboxes is the only sandboxing solution that allows coding agents to build and run Docker containers while remaining isolated from the host system."

How It Works — Architecture

Each sandbox is a microVM with a complete private environment:

┌────────────────────────────────────────────────────────────┐ │ YOUR HOST │ │ │ │ ~/.docker/sandboxes/sandboxd.sock ←── sandboxd daemon │ │ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ microVM (per sandbox) │ │ │ │ │ │ │ │ Your project dir (mounted at same path) │ │ │ │ /Users/you/project ←→ VirtioFS passthrough │ │ │ │ │ │ │ │ Private Docker daemon (own socket) │ │ │ │ ~/.docker/sandboxes/vm/<n>/docker.sock │ │ │ │ │ │ │ │ AI Coding Agent (Claude Code, Codex, etc.) │ │ │ │ └── Builds images, runs containers safely │ │ │ │ │ │ │ └─────────────────┬────────────────────────────────┘ │ │ │ HTTP/HTTPS only │ │ ▼ │ │ Network Proxy host.docker.internal:3128 │ │ ├── Enforces allow/deny domain lists │ │ ├── Injects API keys (agent never possesses them) │ │ ├── Blocks: raw TCP/UDP, cloud metadata endpoints │ │ └── Blocks: private networks, localhost │ │ │ └────────────────────────────────────────────────────────────┘

Key design decisions:

  • Private Docker daemon — the agent can use Docker freely, but it only sees containers inside its own VM. Your host containers are completely invisible.
  • Same absolute path — your project at /Users/you/project appears at /Users/you/project inside the VM too. No path translation needed in the agent's prompts.
  • API keys never enter the VM — the network proxy intercepts outbound calls to Anthropic/OpenAI/etc. and injects credentials there. Even if the agent is compromised or the VM is breached, it cannot exfiltrate your keys.
  • State persists — unlike containers, sandboxes survive until you explicitly remove them. Installed packages are still there when you restart.

Hypervisor Technology

Docker does not publicly name the underlying microVM technology. From platform behavior and the reverse engineering done by Rivet (see sources), each platform uses its native hypervisor:

Understanding how strong the isolation is relative to other approaches:

Docker Sandboxes sits at the fourth level. Stronger than any container-based approach. Not quite as strong as a fully bare-metal-level VM (no hardware passthrough, more restricted I/O model), but the trade-off is justified: it runs in seconds, works on your laptop, and integrates directly with the coding agents you're already using.

Network Policies

One of the most important features — and least visible. The network proxy runs on your host and all sandbox outbound traffic routes through it. You pick a policy mode when you first set up:

ModeDefault behaviorBest for
Open All traffic allowed Trusted code, maximum flexibility
Balanced (default) Default deny, pre-allowed: GitHub, npm, PyPI, crates.io, AI APIs, common CDNs Most developers — good trade-off
Locked Down All traffic blocked unless explicitly allowed Maximum security, untrusted agents

Always blocked regardless of policy: raw TCP/UDP (only HTTP/HTTPS proxied), private networks (10.x, 172.16.x, 192.168.x), cloud metadata services (169.254.169.254), and host localhost.

Important limitation: The network proxy cannot prevent data exfiltration to allowed domains. If GitHub is on the allow-list, the agent could theoretically push sensitive files to a GitHub repo. Sandboxes prevent filesystem and host system compromise — they don't govern whether the agent's decisions are appropriate.

CLI Reference — Both Interfaces

Two separate tools exist. sbx is the standalone CLI (no Docker Desktop required, ships as of March 31, 2026). docker sandbox is the Docker Desktop plugin (requires Desktop 4.58+).

Install sbx

# macOS
brew install docker/tap/sbx

# Windows
winget install -h Docker.sbx

# Authenticate (uses Docker OAuth)
sbx login

Core workflow

# Create a sandbox and launch Claude Code in your project
sbx run claude ~/my-project

# With an initial task prompt
sbx run claude ~/my-project -- "Fix all TypeScript errors in src/"

# Pass flags to the agent (e.g. --continue last session)
sbx run claude ~/my-project -- --continue

# List all your sandboxes
sbx ls

# Open an interactive shell inside a running sandbox
sbx exec my-sandbox -it bash

# Stop a sandbox (state preserved, can restart)
sbx stop my-sandbox

# Delete a sandbox (removes VM, NOT your project files)
sbx rm my-sandbox

# Nuke everything
sbx reset

Network and secrets

# Store an API key securely (never goes inside the VM)
sbx secret set -g anthropic     # prompts for ANTHROPIC_API_KEY
sbx secret set -g openai        # prompts for OPENAI_API_KEY

# View what network requests were blocked
sbx policy log my-sandbox

# Allow an additional domain
sbx policy allow network my-sandbox --domain example.com

# Bypass proxy for a host (certificate pinning workaround)
sbx policy --bypass-host api.example.com

# Expose a port from sandbox to host (e.g. dev server)
sbx ports my-sandbox --publish 3000

docker sandbox equivalents (Docker Desktop plugin)

docker sandbox create claude ~/my-project
docker sandbox run <sandbox-name>
docker sandbox exec <sandbox-name>
docker sandbox ls
docker sandbox rm <sandbox-name>
docker sandbox stop <sandbox-name>
docker sandbox inspect <sandbox-name>
docker sandbox network
docker sandbox reset
docker sandbox version

Supported agents

claude codex copilot gemini kiro opencode docker-agent shell

The shell subcommand creates a blank microVM with no agent — useful for running arbitrary isolated workloads or testing the environment itself.

Examples You Can Run Right Now

1. Verify the sandbox is isolated from your host Docker

# On your host, start a recognizable container
docker run -d --name sentinel nginx:alpine

# Create a blank sandbox (shell mode = no agent, just an isolated shell)
sbx run shell ~/any-empty-dir

# Inside the sandbox, try to see host containers:
docker ps
# ✓ Result: empty — your 'sentinel' container is completely invisible

# Try to reach the host Docker daemon:
ls ~/.docker/sandboxes/
# ✓ Result: not accessible — different filesystem

# Verify you have your OWN Docker daemon:
docker info | grep "Docker Root Dir"
# ✓ Shows a path inside the VM, not /var/lib/docker on your host

# Clean up
docker stop sentinel && docker rm sentinel

2. Test file sync (project visible at same path)

# Create a test project
mkdir ~/sbx-test && echo "hello from host" > ~/sbx-test/probe.txt

# Launch a shell sandbox on it
sbx run shell ~/sbx-test

# Inside the sandbox:
cat ~/sbx-test/probe.txt          # ✓ "hello from host" — same absolute path
echo "hello from sandbox" >> ~/sbx-test/probe.txt

# Back on your host (changes sync immediately):
cat ~/sbx-test/probe.txt          # ✓ Shows both lines

3. Test network policies

# Inside the sandbox shell:
curl -s https://api.github.com/zen     # ✓ Allowed in Balanced mode
curl -s https://registry.npmjs.org/    # ✓ Allowed
curl -s https://169.254.169.254/       # ✗ Always blocked (AWS metadata)
curl -s http://192.168.1.1/            # ✗ Blocked — private network

# On your host, watch what was blocked:
sbx policy log my-sandbox

4. Run Docker-in-Docker safely

# Inside the sandbox — build and run a container (your own private daemon):
cat > Dockerfile << 'EOF'
FROM alpine:latest
CMD ["echo", "I am a container inside a sandbox"]
EOF

docker build -t inner-test .
docker run --rm inner-test
# ✓ Works! And none of this appears on your host's docker ps

5. Run Claude Code on a real project

# Store your API key once (never goes in the VM)
sbx secret set -g anthropic

# Run Claude Code against your project
sbx run claude ~/my-project

# With a specific task
sbx run claude ~/my-project -- "Add unit tests for all public functions in src/parser.rs"

# The agent can:
#   ✓ Read and write files in ~/my-project
#   ✓ Run cargo build, npm install, pip install, etc.
#   ✓ Build and run Docker containers
#   ✗ See your host Docker containers
#   ✗ Access ~/.ssh, ~/.aws, or any other host directory
#   ✗ Access cloud metadata endpoints
#   ✗ Make raw TCP connections

6. Inspect the sandbox internals (Rivet reverse-engineering approach)

# The sandboxd daemon exposes an HTTP API on its unix socket
# List all VMs:
curl --unix-socket ~/.docker/sandboxes/sandboxd.sock \
  http://localhost/vm

# Each VM has its own Docker socket:
ls ~/.docker/sandboxes/vm/

# You can use the sandbox's private Docker daemon directly:
DOCKER_HOST=unix://$HOME/.docker/sandboxes/vm/<n>/docker.sock \
  docker ps -a

7. Locked-down sandbox for maximum safety

# Create a sandbox then enforce strict network policy
sbx create shell ~/untrusted-project

# Lock it down — deny everything by default
sbx policy set my-sandbox --mode locked-down

# Only allow what your project actually needs:
sbx policy allow network my-sandbox --domain api.anthropic.com
sbx policy allow network my-sandbox --domain github.com

# Now run it
sbx run my-sandbox

Known Limitations & Bugs

LIMIT
Linux is experimental and single-user only. Uses container-based isolation (not a real microVM) and only works for UID 1000. Not suitable for multi-user servers or CI environments.
LIMIT
No raw TCP/UDP outbound. Only HTTP/HTTPS is proxied. Breaks anything that needs raw socket connections: ADB (Android), custom game servers, MQTT, etc.
LIMIT
One sandbox per workspace directory. You can't run two sandboxes against the same project path simultaneously. Remove the old one first.
LIMIT
Agent governance is not included. The sandbox prevents filesystem and kernel escapes, but it can't stop the agent from making inappropriate authorized decisions — e.g., deleting files it shouldn't, merging PRs, overwriting configs. That's a separate problem.
BUG
Clock drift after sleep/wake. The VM clock drifts when your laptop sleeps, causing HTTPS certificate errors. Fix: sbx stop <name> then sbx run <name>.
BUG
Stale git worktrees on removal. After sbx rm, git worktrees may remain. Manual cleanup: git worktree remove .sbx/<name> and git branch -D <branch>.
BUG
Windows: multiple sandbox crashes. Launching too many simultaneously can crash. Fix: kill all docker.openvmm.exe processes and restart.
BUG
Windows MSI upgrade bug. The MSI installer cannot update existing installs of 4.59+. You must fully uninstall first (which deletes all sandbox data).
NOTE
Certificate pinning services break. Some APIs use certificate pinning and reject the proxy's MITM cert. Workaround: sbx policy --bypass-host <domain>.
NOTE
OrbStack incompatibility. docker sandbox doesn't work with OrbStack as the Docker backend (OrbStack feature request #2295 is open).

What Docker Sandboxes Is NOT

Common misconception: Docker Sandboxes is not a general-purpose container security hardening feature. It won't make your web server containers more secure. It won't help you run untrusted third-party images safely in production. It won't replace Kubernetes security policies.

For those use cases, the right tools are still:

  • seccomp profiles — filter syscalls inside containers
  • AppArmor / SELinux — mandatory access control policies
  • gVisor (runsc) — intercept all syscalls in userspace, used by Google Cloud Run
  • Kata Containers / Firecracker — VM-per-container, used by AWS Lambda and Fly.io
  • rootless containers — run Docker daemon and containers without root

Docker Sandboxes solves exactly one problem very well: safely running AI coding agents against your codebase. That's it.

Quick Reference — What to Use When

ScenarioUseWhy
Running Claude Code / Codex on my project Docker Sandbox (sbx) Exactly what it's built for
