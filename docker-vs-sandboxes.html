What Is Docker Sandbox?

Docker Sandboxes is a feature from Docker (introduced late 2025, matured in early 2026) that runs AI coding agents in isolated microVM environments. It is fundamentally different from running an agent in a regular Docker container.

The core insight Docker acted on: AI agents like Claude Code, GitHub Copilot, and Google Gemini CLI need to be able to do dangerous things safely — install packages, rewrite files, spawn nested Docker builds — without touching your host system, your credentials, or your other projects. Containers share the host kernel and cannot safely host a nested Docker daemon. MicroVMs solve this at the hardware level.

One-sentence definition A Docker Sandbox is a full virtual machine — with its own kernel, its own Docker daemon, and its own filesystem — that you can spin up in seconds, give to an AI agent to work in, then destroy when done.

Each sandbox:

  • Has its own dedicated Linux kernel (not the host's)
  • Has its own private Docker daemon (agents can run docker build, docker compose without touching the host daemon)
  • Mounts your workspace directory at the same absolute path as on the host
  • Routes outbound network through a proxy that can enforce allow/deny lists
  • Is destroyed cleanly with a single command — workspace files persist, everything else is gone

Architecture — MicroVM vs. Containers

Why Not Just Containers?

Regular Docker containers share the host kernel. This means:

  • An agent running in a container can see and interact with the host's Docker daemon if the socket is mounted
  • Docker-in-Docker either compromises isolation (privileged mode) or requires complex setup
  • Kernel-level exploits can escape the container to the host
  • Files like ~/.ssh and ~/.aws can leak if the agent explores bind mounts

What Docker Sandboxes Uses Instead

Docker Sandboxes uses Firecracker microVMs on top of the host OS's native hypervisor. Firecracker is AWS's open-source VMM written in Rust (~83,000 lines), originally designed for AWS Lambda and Fargate. It boots in ~125ms with less than 5 MiB of memory overhead per VM — faster and lighter than any traditional VM while providing true hardware isolation.

OSHypervisor LayerNotes
macOS (Apple Silicon)Firecracker + Apple Virtualization.frameworkUses Apple's native hypervisor (Hypervisor.framework under the hood)
Windows 11Firecracker + Windows Hypervisor Platform (Hyper-V)Must be enabled; see OS support section
Linux (experimental)Firecracker + KVMSingle-user (UID 1000) only; Docker Desktop required

Sandbox Internals

Inside each sandbox

  • Dedicated Linux kernel (not host's)
  • Private Docker daemon
  • Workspace mounted at identical host path
  • Isolated filesystem and network stack
  • Pre-installed AI agent (Claude, Codex, etc.)

On the host

  • MicroVM managed by Docker Desktop
  • Outbound HTTP/HTTPS proxy for policy enforcement
  • Credential injection (API keys never enter sandbox filesystem)
  • Port forwarding for services the agent runs
Key differentiator Docker Sandboxes is the only solution that lets AI agents do full Docker-in-Docker operations while remaining completely isolated from the host Docker daemon. Regular container isolation cannot achieve this safely.

Release Timeline

Late 2025
Docker Desktop — Container-Based Prototype
Early experimental sandbox implementation using privileged containers (not VMs). Introduced the concept but had weaker isolation — agents still shared the host kernel. This version was superseded by the Firecracker redesign in 4.58.
January 26, 2026
Docker Desktop 4.58 — Firecracker MicroVM Launch Architecturally significant
Complete rewrite using Firecracker microVMs. Each sandbox gets a dedicated kernel, private Docker daemon, and hardware-level isolation. Introduced the sbx standalone CLI. Announced in blog post January 30, 2026.
February 2, 2026
Docker Desktop 4.59 — Linux, WSL2, Shell Mode, Caching
Experimental Linux support (single-user, UID 1000, KVM-based). WSL2 support on Windows. Shell agent template (blank VM, no AI). OpenCode agent. Automated image caching for faster subsequent starts. Mount multiple workspaces.
March 31, 2026
sbx CLI v0.21.0 — "YOLO Mode, Safely" Blog Post
Standalone CLI independent of Docker Desktop. sbx rm --all. sbx policy reset. Kiro agent support. .deb/.rpm packages for Linux. Docker publishes "Run Agents in YOLO Mode, Safely" blog post confirming Firecracker architecture.

CLI Reference

There are two CLI surfaces:

  • sbx — standalone CLI, more features, does not require Docker Desktop to be running
  • docker sandbox — built-in to Docker Desktop, subset of commands

Installing sbx

# macOS (Apple Silicon)
brew install docker/tap/sbx

# Windows 11
winget install -h Docker.sbx

# Linux (.deb / .rpm packages)
# Download from: https://github.com/docker/sbx-releases

Authentication

sbx login           # Opens browser for Docker OAuth
                    # Choose network policy: Open / Balanced / Locked Down

Creating and Running Sandboxes

# Create + immediately run (most common usage)
sbx run claude                     # Claude Code agent in current directory
sbx run claude -- --continue       # Pass args to the agent (-- separates sbx args)
sbx run codex                      # OpenAI Codex agent
sbx run gemini                     # Google Gemini CLI agent
sbx run copilot                    # GitHub Copilot agent
sbx run kiro                       # Amazon Kiro agent
sbx run opencode                   # OpenCode agent
sbx run shell                      # Blank VM (no AI pre-installed)
sbx run docker-agent               # General Docker agent

# Create without attaching (background)
sbx create docker-agent .          # Current directory
sbx create docker-agent /my/proj   # Specific path

# Attach to an existing sandbox
sbx run my-sandbox-name

Managing Sandboxes

sbx ls                              # List all sandboxes (NOT visible in docker ps)
sbx exec <name> bash               # Shell into a running sandbox
sbx stop <name>                     # Halt VM, preserve state
sbx rm <name>                       # Remove (destroys VM and all installed state)
sbx rm sandbox1 sandbox2           # Remove multiple sandboxes
sbx rm --all                        # Remove all sandboxes (v0.21.0+)
sbx version                        # Show CLI and daemon version

# Memory override (workaround for 4GB cap bug in docker sandbox)
sbx run claude --memory 8g

Docker Desktop Extra Commands

docker sandbox inspect <name>      # Detailed sandbox info
docker sandbox stop <name>         # Stop without removing
docker sandbox reset               # Remove ALL sandboxes and clean up state

# Save a snapshot as a reusable template (becomes a Docker image)
docker sandbox run shell           # 1. start a blank sandbox
# install tools inside...
docker sandbox save my-java-env    # 2. save as template
docker sandbox run -t my-java-env claude  # 3. reuse with any agent

Port Management

sbx ports <name>                           # List all ports for a sandbox
sbx ports <name> --publish 8080           # Publish port → ephemeral host port
sbx ports <name> --publish 3000:8080      # Publish with specific host port
sbx ports <name> --unpublish 3000:8080   # Unpublish a port

Network Policy

sbx policy log                             # View network access log
sbx policy allow network "*.npmjs.org"    # Allow a domain
sbx policy allow network "10.0.0.0/8"    # Allow a CIDR range
sbx policy allow network "**"            # Allow all traffic (Open mode)
sbx policy reset                          # Restore default policy (v0.21.0+)

docker sandbox (built-in Docker Desktop commands)

docker sandbox run AGENT [PATH]
docker sandbox ls
docker sandbox exec -it <name> bash
docker sandbox rm <name>
docker sandbox reset                      # Remove all sandboxes
docker sandbox network proxy <name> --policy allow|deny
docker sandbox network log
Important Sandboxes are VMs, not containers. They do not appear in docker ps. Use sbx ls or docker sandbox ls to see running sandboxes.

Supported

  • Intel Mac: unconfirmed; docs say "Apple Silicon" explicitly

Supported
  • Windows 11 x86_64 only
  • ARM Windows: not supported
  • Requires: Windows Hypervisor Platform (not full Hyper-V — enable via PowerShell)
  • WSL2 supported since Docker Desktop 4.61
  • Docker not default in Windows sandbox — use docker-agent template
  • Install: winget install -h Docker.sbx

Linux

Experimental
  • Since Docker Desktop 4.61 (Feb 18, 2026)
  • Single-user only (UID 1000)
  • Requires KVM support
  • Multi-user not yet supported
  • Nested virtualization not supported
  • Install: .deb/.rpm packages from sbx-releases

Enabling Windows Hypervisor Platform

# Run in elevated PowerShell (as Administrator)
Enable-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform

# Verify it's enabled
Get-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform

When to Use Sandbox vs. Container

✦ Use Docker Sandbox when…

  • Running AI coding agents autonomously (unattended)
  • The agent needs to run docker build / docker compose internally
  • You want the agent isolated from ~/.ssh, ~/.aws, and other projects
  • You want complete host isolation (cannot escape to host kernel)
  • You want disposable environments (reset in one command)
  • Running multiple agents simultaneously without shared state
  • Network policy control over what the agent can access

◆ Use Docker Container when…

  • Running trusted, well-understood workloads (web apps, databases)
  • Maximum density and performance are required
  • No nested Docker daemon needed
  • Long-lived, resource-constrained services
  • CI/CD pipelines with trusted code
  • Microservices and production workloads
  • You need docker ps visibility and standard tooling

Decision Flowchart

Is an AI agent making autonomous file or system changes?
├── YES → Is it running trusted, bounded code?
│         ├── YES → Container may be fine
│         └── NO  → ✦ Use Docker Sandbox
└── NO  → Does it need nested Docker capabilities?
          ├── YES → ✦ Use Docker Sandbox
          └── NO  → ◆ Use Docker Container

Security Model

What Docker Sandbox Protects

Protected ResourceHow
Host filesystem~/.ssh, ~/.aws, and all files outside the workspace are invisible inside the sandbox
Host Docker daemonCompletely isolated — agent cannot see host containers, images, or volumes
System packagesAgent installs don't touch the host OS; destroyed when sandbox is removed
Cross-sandbox stateNo shared images, layers, filesystems, or networks between sandboxes
Network trafficOutbound filtered through configurable proxy (Balanced, Locked Down, Open modes)
API credentialsKeys stored on host; proxy injects into outbound API requests — never written inside sandbox

Network Policy Modes

ModeBehaviorUse Case
OpenUnrestricted internet accessTrusted agent, development, speed over security
BalancedDefault-deny with common dev sites pre-allowed (npm, pypi, GitHub, etc.)Default recommended mode for most users
Locked DownAll traffic blocked unless explicitly permittedMaximum security; add domains one by one
What Docker Sandbox Does NOT Protect
  • Your workspace files — the agent has full read/write access to your mounted project. It can delete everything.
  • API key usage — if the agent has your OpenAI/Anthropic key, it can call APIs as much as it wants within the sandbox
  • Network exfiltration of workspace data — unless you use Locked Down mode, the agent can POST your code to external servers

Performance

MetricDocker ContainerDocker Sandbox (Firecracker)
Cold start time<100ms~125ms (Firecracker VM boot) + agent image pull
Warm start (cached)<100ms28–300ms (snapshot/restore path; caching added 4.59)
VMM memory overheadNear-zero<5 MiB (Firecracker VMM itself)
Per-sandbox total memory~50–100 MiB per containerVM kernel + daemon: ~100–200 MiB additional
CPU overhead at runtimeNear-zero (<1%)Single-digit % (virtio / VM boundary)
Filesystem I/ONative or overlay FSvirtio-fs; slightly slower for I/O-heavy workloads
NetworkNative bridge/overlayProxied; minor overhead for policy enforcement
Why Firecracker is fast enough Firecracker was purpose-built for ephemeral workloads (AWS Lambda). Its 125ms boot and <5 MiB overhead make it vastly faster and lighter than traditional VMs (which take seconds to boot and use hundreds of MiB). The performance gap between a Firecracker sandbox and a regular container is small enough to be irrelevant for AI agent workloads that run for minutes or hours.
Caching note Docker Desktop 4.61 added automated image caching for sandboxes. On subsequent sbx run calls for the same agent, the sandbox starts within seconds because the agent image is already cached.

Hands-On Examples

Example 1 — Run Claude Code on Your Project

# Install sbx
brew install docker/tap/sbx   # macOS

# Authenticate (one-time)
sbx login
# → opens browser, choose "Balanced" network policy

# Navigate to your project
cd ~/my-project

# Launch Claude Code in a sandbox
sbx run claude
# → creates VM, mounts ~/my-project at the same path inside
# → opens Claude Code interactive session
# → your ~/.ssh and ~/.aws are NOT visible inside

# When done, list and remove the sandbox
sbx ls
sbx rm <sandbox-name>

Example 2 — Non-Interactive Agent with Port Forwarding

# Create sandbox, run agent to build a web app, expose the dev server
sbx create claude /my/webapp

# See what's running
sbx ls
# NAME             AGENT   STATUS    WORKSPACE
# vibrant-panther  claude  running   /my/webapp

# Publish the dev server port (agent started something on 3000)
sbx ports vibrant-panther --publish 3000
# → http://localhost:49152  (ephemeral port)

# Shell in to inspect
sbx exec vibrant-panther bash

# Done — clean up everything
sbx rm vibrant-panther

Example 3 — Blank Shell Sandbox for Experimentation

# Shell sandbox = blank VM, you install what you need
sbx run shell

# Inside the sandbox VM
apt-get install -y nodejs npm
npm install -g my-dangerous-package   # safe — host is unaffected
docker build -t test-image .          # nested Docker works
docker run --rm test-image echo hello

# Exit and destroy (Ctrl+D or exit)
# Then on the host:
sbx rm --all   # clean slate

Example 4 — Locked Down Network for Sensitive Code

# Set locked-down policy before running
sbx login   # choose "Locked Down" during login

# Or change policy after login
# Start a sandbox
sbx run claude

# In another terminal — allow only what the agent needs
sbx policy allow network "*.anthropic.com"
sbx policy allow network "*.npmjs.org"
sbx policy allow network "*.pypi.org"

# View what was accessed
sbx policy log

# Reset to defaults
sbx policy reset

Example 5 — Windows 11 Setup

# Step 1: Enable hypervisor (elevated PowerShell)
Enable-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform
# Restart required

# Step 2: Install Docker Desktop 4.58+
# Download from https://www.docker.com/products/docker-desktop/

# Step 3: Install sbx CLI
winget install -h Docker.sbx

# Step 4: Login and run
sbx login
sbx run claude

# From WSL2 (Windows 11, Docker Desktop 4.61+)
# sbx is also accessible from inside WSL2 terminal
