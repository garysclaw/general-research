MATLAB Integration Approaches: A Complete Guide

There are at least eleven distinct ways to connect MATLAB to the rest of your software stack — and they differ radically in licensing requirements, startup overhead, deployment complexity, and what you can actually do with them. This is a thorough, opinionated breakdown of every approach that matters in 2025/2026, from the official Engine API to ZMQ socket bridges, with honest assessments of where each one breaks down.

Quick Comparison

jupyter-matlab-proxy Full MATLAB Notebook Low Interactive notebooks, teaching, JupyterHub

Licensing Reality

Every approach except MATLAB Compiler+MCR deployment, GNU Octave, and Python still requires an active MATLAB license to run code. The Compiler only moves the license requirement to build time — end users get the free MCR, but you still pay for MATLAB and MATLAB Compiler toolbox to build in the first place.

1. MATLAB Engine API for Python (matlab.engine)

matlab.engine — Official MathWorks Python API
Official License Required In-Process
The MATLAB Engine API for Python starts a full MATLAB process from within Python and lets you call any MATLAB function, transfer arrays, and read/write the MATLAB workspace directly. Released as a proper PyPI package (matlabengine) since MATLAB R2022b, it is the most direct way to call MATLAB from Python without intermediate files or sockets.
Links: PyPI: matlabengine · GitHub · Limitations docs · Python version compatibility
Pros
  • Official, MathWorks-supported
  • Call any MATLAB function by name
  • Read/write MATLAB workspace variables
  • Async startup via background=True
  • Connect to shared MATLAB sessions
  • Available on PyPI since R2022b
Cons
  • Full MATLAB license required on every machine
  • Startup time: 10–30s for a fresh MATLAB process
  • Version pinning: matlabengine==9.13.7 must match your MATLAB release exactly
  • 2 GB data size limit per call
  • No remote connections — localhost only
  • No Python keyword arguments — positional only
  • Recursive data structures not supported
  • Python 2.x dropped since R2023a

How it works

matlab.engine.start_matlab() spawns a MATLAB process and returns an engine handle. Every subsequent call blocks Python until MATLAB returns. The startup cost is paid once; subsequent calls are fast. Shared engines let multiple Python scripts connect to the same already-running MATLAB session to amortize startup time. 

import matlab.engine

# Start synchronously (blocks ~15–30 seconds)
eng = matlab.engine.start_matlab()

# Or start in background, do other work, then wait
future = matlab.engine.start_matlab(background=True)
eng = future.result()

# Call any MATLAB function
result = eng.sqrt(16.0)            # returns 4.0
val = eng.eval("pi * 2", nargout=1)  # eval a string

# Transfer numpy-like arrays
import matlab
arr = matlab.double([[1, 2, 3], [4, 5, 6]])
result = eng.sum(arr)

# Connect to a named shared engine (already running)
eng2 = matlab.engine.connect_matlab('my_session')

eng.quit()
Version Pinning Trap

The matlabengine PyPI package version must exactly match your installed MATLAB release. R2024b ships engine version 24.2. If you upgrade MATLAB, you must also update the Python package — and if your environment pins it via requirements.txt, the mismatch will silently fail at import time with a cryptic error.

2. jupyter-matlab-proxy — Browser-Based MATLAB in Jupyter

jupyter-matlab-proxy
Official MathWorks License Required JupyterHub Ready
MathWorks' official Jupyter integration package. It does two distinct things: (1) provides an official MATLAB kernel for Jupyter notebooks, and (2) proxies the full MATLAB desktop GUI into a browser tab directly from JupyterLab. You can write .ipynb notebooks with MATLAB cells and also open the full MATLAB IDE — including App Designer, Simulink, and the command window — without leaving your browser.
Links: GitHub · PyPI · MathWorks product page · Official announcement blog
Pros
  • Official MathWorks package — actively maintained
  • Full MATLAB IDE in browser (not just a kernel)
  • Works with JupyterHub, single-user servers, and cloud deployments
  • Windows support added in v0.6.0
  • Licensing UI built-in — prompts on first execution
  • Multiple notebooks share one MATLAB process by default
  • Active release cadence: v0.16.4 shipped March 2025
Cons
  • Full MATLAB license (or network license) required
  • MATLAB must be installed and on PATH
  • Heavier than a bare kernel — proxies full MATLAB process
  • Shared workspace across notebooks is confusing in multi-user setups
  • Simulink UI requires VNC or reference architecture setup
  • Some MATLAB commands unsupported in kernel mode

Kernel mode vs. desktop proxy mode

These are two separate capabilities bundled in one package. In kernel mode, MATLAB executes cells in a Jupyter notebook like any other language kernel — outputs appear inline, plots render in the notebook. In desktop proxy mode, clicking "Open MATLAB" launches the full MATLAB desktop in a new browser tab, proxied through Jupyter's server. Desktop mode is significantly heavier but gives you access to every MATLAB GUI feature including App Designer and Simulink (without needing a separate VNC setup on Linux). 

# Install
pip install jupyter-matlab-proxy

# Launch JupyterLab — MATLAB kernel appears in the kernel picker
jupyter lab

# In a MATLAB notebook cell:
x = linspace(0, 2*pi, 100);
y = sin(x);
plot(x, y);
title('Sine wave');

# Use %%matlab new_session to get a dedicated MATLAB process
# (instead of the shared default)
JupyterHub Deployment

For JupyterHub, install jupyter-matlab-proxy in the Docker image or conda environment that JupyterHub spawns for each user. Each user's MATLAB session is isolated at the Jupyter server level. You will need a concurrent-user network license or a license server with enough seats to cover simultaneous users.

3. Community MATLAB Kernels — imatlab and matlab_kernel

imatlab + matlab_kernel (Community Kernels)
Community License Required Jupyter Kernel
Two community-maintained alternatives to the MathWorks official kernel. imatlab (by the imatlab GitHub org) uses the MATLAB Engine API directly and supports interactive Plotly graphics with shared engine connections. matlab_kernel (by Calysto) is built on MetaKernel and provides a rich magics system and configurable plot output settings.
Links: imatlab GitHub · matlab_kernel GitHub
Pros
  • imatlab: interactive Plotly graphics (not just static images)
  • imatlab: shared engine support via IMATLAB_CONNECT
  • matlab_kernel: full MetaKernel magics (%lsmagic)
  • matlab_kernel: configurable plot format, resolution, backend
  • Both are lighter than jupyter-matlab-proxy
  • History shared with MATLAB's native History.xml (imatlab)
Cons
  • Community maintained — lower support guarantees
  • Both require MATLAB Engine API (full license)
  • imatlab: async timer output completely unsupported
  • imatlab: MATLAB debugger cleared before each execution
  • imatlab: Windows timing issues with pause()
  • No browser-based MATLAB desktop access
  • MathWorks' official kernel is now preferred for new setups
# Install imatlab
pip install imatlab
python -m imatlab install  # registers the kernel with Jupyter

# Install matlab_kernel
pip install matlab_kernel
python -m matlab_kernel install

# Connect imatlab to a named shared MATLAB engine
# (in MATLAB: matlab.engine.shareEngine('mySession'))
IMATLAB_CONNECT=mySession jupyter lab

# matlab_kernel plot magic
%plot -f svg -r 150 -s 800,600
Which kernel to choose?

For new setups: use jupyter-matlab-proxy — it is official, actively maintained, and includes the browser desktop. Use imatlab if you need interactive Plotly graphics in notebooks. Use matlab_kernel if you need the MetaKernel magics system or fine-grained plot control and can't use the proxy.

4. Subprocess / Batch Mode

matlab -batch — Non-Interactive Automation
Simple License Required CI/CD Friendly
Calling the MATLAB executable as a subprocess is the simplest form of integration — no Python package required, no version pinning, no sockets. The -batch flag (R2019a+) runs a script or expression non-interactively, exits with a proper return code on error, and is the correct modern approach for CI/CD and HPC. Older -r "cmd; exit" still works but exits 0 on failure.
Links: batchStartupOptionUsed docs · Batch mode guide
Pros
  • Zero Python dependencies — just subprocess
  • Works in any language or shell script
  • Correct exit codes with -batch
  • -nodisplay/-nosplash reduce overhead on Linux/HPC
  • -nojvm skips JVM for pure computation scripts
  • Easy to integrate into Makefiles, GitHub Actions, Slurm
Cons
  • Full MATLAB startup on every call (~15–30s)
  • No interactive input — headless only
  • Data exchange via files (.mat, CSV, JSON) or stdout parsing
  • stdout parsing is fragile if MATLAB warnings contaminate output
  • Full MATLAB license still required
  • One MATLAB process per subprocess call — expensive at scale
# Modern batch mode (-batch exits non-zero on error)
matlab -nodisplay -nosplash -batch "run('my_script.m')"

# Pass parameters via environment variables or -r
matlab -nodisplay -nosplash -batch "param=42; run('compute.m')"

# In Python via subprocess
import subprocess, json, pathlib

def run_matlab_script(script_path, output_json):
    result = subprocess.run(
        ['matlab', '-nodisplay', '-nosplash', '-batch',
         f"run('{script_path}')"],
        capture_output=True, text=True, timeout=300
    )
    if result.returncode != 0:
        raise RuntimeError(result.stderr)
    return json.loads(pathlib.Path(output_json).read_text())

# In MATLAB script: save results to JSON for Python to read
# result = struct('value', 42, 'status', 'ok');
# fid = fopen('output.json', 'w');
# fprintf(fid, jsonencode(result));
# fclose(fid);

# Guard interactive-only code in batch mode
# if ~batchStartupOptionUsed; userInput = input('Enter value: '); end
-batch vs. -r

The -batch flag was added in R2019a specifically to fix the silent-failure problem: if your script throws an unhandled error, -batch exits with code 1 (detectable in CI). The old -r "cmd; exit" exits 0 regardless. Always prefer -batch in automated pipelines. Use batchStartupOptionUsed inside MATLAB scripts to conditionally skip interactive prompts.

5. MATLAB Compiler + MATLAB Runtime (MCR)

MATLAB Compiler + MCR — License-Free Deployment
No Runtime License Compiler License Required (build) Standalone
MATLAB Compiler (mcc) compiles MATLAB code into standalone executables, shared libraries (C/C++/Java/.NET), or Python packages. End users run against the free MATLAB Runtime (MCR) — no MATLAB license needed at deployment time. Starting in R2024b, you can create minimal custom MCR installers to reduce the historically bloated runtime footprint (typically 3–5 GB).
Links: MATLAB Runtime downloads · About MCR · Licensing docs
Pros
  • End users need no MATLAB license — MCR is free
  • Protects IP: source .m files encrypted in CTF archive
  • Compile to exe, shared library (.so/.dll), Python package
  • R2024b: custom minimal MCR installers available
  • R2024b: GPU libraries no longer required by default
  • Docker containers supported since R2022b
  • Works on Windows, Linux, macOS
Cons
  • Requires MATLAB Compiler toolbox license to build
  • MCR must exactly match the compiler version used
  • MCR installer still ~1–5 GB depending on toolboxes used
  • Cold startup adds ~10–20s for MCR initialization
  • Not all MATLAB toolboxes are compilable (some require live license)
  • Dynamic code (eval, load from paths) may fail at runtime
  • Debugging compiled apps is significantly harder

Compilation targets

The mcc compiler generates several types of artifacts depending on the flags used:

  • Standalone executable (mcc -m): a binary that bundles all dependencies into a CTF (Component Technology File) archive and calls into MCR.
  • C/C++ shared library (mcc -l): a .so or .dll callable from C/C++ code.
  • Python package (compiler.build.pythonPackage): wraps MATLAB functions in a Python wheel, callable via a thin Python API backed by MCR.
  • Java package (mcc -W java): a JAR with callable Java classes.
  • .NET assembly: a DLL for .NET/C# consumers.
% In MATLAB: compile a function to a standalone exe
mcc -m my_function.m -o my_app

% Compile to a Python package (MATLAB R2021b+)
compiler.build.pythonPackage('my_function.m', ...
    'PackageName', 'MyMatlabLib', ...
    'OutputDir', './my_matlab_lib_pkg');

% Build a custom minimal MCR installer (R2024b+)
compiler.runtime.customInstaller('my_app.ctf', ...
    'OutputDir', './installer');
# Python: call a compiled MATLAB package via MCR
import sys
sys.path.append('/path/to/my_matlab_lib_pkg')

import my_matlab_lib
import matlab.engine

lib = my_matlab_lib.initialize()
result = lib.my_function(3.14, nargout=1)
lib.terminate()
Version Pinning Is Absolute

MCR version pinning is stricter than the Engine API. A compiled app built with R2024a will only run against the R2024a MCR — not R2024b, not R2023b. This makes long-term maintenance painful: any MATLAB upgrade requires recompilation and redeployment. Plan your release cadence accordingly.

6. MATLAB Production Server

MATLAB Production Server — Enterprise REST Deployment
Enterprise License REST API Scalable
MATLAB Production Server (MPS) hosts compiled MATLAB functions as REST API endpoints. Client applications in any language (Python, Java, .NET, JavaScript) call these endpoints over HTTP/HTTPS. MPS handles concurrency, scaling, and health monitoring. As of R2024b it also supports deploying MATLAB functions as MCP (Model Context Protocol) tools callable from AI agents and chat applications.
Links: MPS product page · REST API docs · OpenAPI integration (GitHub)
Pros
  • Clients need no MATLAB knowledge or license
  • REST API works from any language or HTTP client
  • Horizontal scaling via load balancer
  • Kubernetes/Docker support (AWS, Azure, GKE)
  • Built-in Prometheus metrics and health endpoints
  • Async request mode supported
  • MCP tool support for AI agents (R2024b+)
Cons
  • Separate, expensive MPS license required
  • MATLAB Compiler SDK required to build deployable archives
  • Stateless: no persistent workspace between calls
  • Complex to operate for small teams
  • Functions must be pre-compiled — no dynamic MATLAB execution
  • Heavy infrastructure overhead for simple use cases
% Step 1: Compile your function for MPS
compiler.build.productionServerArchive('my_algorithm.m', ...
    'ArchiveName', 'my_archive');

# Step 2: Call via REST from Python
import requests, json

url = "http://mps-server:9910/my_archive/my_algorithm"
payload = {
    "nargout": 1,
    "rhs": [{"mwtype": "double", "mwdata": [3.14]}]
}
resp = requests.post(url, json=payload)
result = resp.json()["lhs"][0]["mwdata"]

7. MATLAB Web App Server

MATLAB Web App Server — Browser-Hosted MATLAB Apps
Enterprise License No MATLAB for Users CTF Deploy
MATLAB Web App Server lets you host apps built in App Designer as interactive web apps accessible from any browser — no MATLAB installation required for end users. Apps are packaged as CTF files using the Web App Compiler and uploaded to the server. R2025a adds direct deploy-to-server from within MATLAB. This is the right tool when you want to expose a GUI tool to non-technical users inside a firewall.
Links: Product page · Web Apps docs · Deploy a Web App
Pros
  • End users need only a browser — no MATLAB
  • Supports OpenID Connect and LDAP authentication
  • Multiple MCR versions supported simultaneously
  • Docker containers supported (R2022b+)
  • No repackaging needed when upgrading MATLAB Runtime
  • Secrets management for API keys and passwords
Cons
  • Requires MATLAB Web App Server license (separate product)
  • Designed for trusted intranet, not public internet
  • Apps must be built in App Designer — no script-based apps
  • No programmatic API: output is always HTML/browser UI
  • Web App Compiler + MATLAB Compiler required to build
  • Not a replacement for a REST API

8. GNU Octave — Open-Source MATLAB-Compatible

GNU Octave
Free / GPL ~95% Compatible No Simulink
GNU Octave is a free, open-source scientific programming language with syntax highly compatible with MATLAB. It runs most MATLAB scripts without modification and integrates with Python via the oct2py bridge library. In 2025, Octave remains the go-to choice for license-free MATLAB-compatible computation — particularly in CI/CD pipelines where a full MATLAB seat is not available.
Links: GNU Octave website · Differences from MATLAB · oct2py GitHub
Pros
  • Completely free, GPL licensed
  • Runs most MATLAB scripts without modification
  • oct2py provides Python bridge similar to matlab.engine
  • Read/write .mat files
  • Octave Forge packages for additional functionality
  • Works in CI/CD without any license server
  • Active community development
Cons
  • No Simulink equivalent — block-diagram modeling not possible
  • JIT compiler is incomplete — loops can be 100x slower
  • Some MATLAB toolboxes have no Octave Forge equivalent
  • Octave-specific syntax extensions can break MATLAB portability
  • No MATLAB v7.3 HDF5 .mat file compatibility
  • Limited commercial support options
  • GUI less polished than MATLAB desktop

Key compatibility gaps in 2025

Octave parses roughly 95% of real-world MATLAB scripts without changes. The gaps that matter most: Simulink (no equivalent), the Statistics and Machine Learning Toolbox (only partially covered by Octave Forge), and Stateflow (no equivalent). The JIT compiler gap is critical for loop-heavy code — vectorize everything or accept a significant performance penalty. 

# Install oct2py for Python-Octave bridge
pip install oct2py

from oct2py import Oct2Py

oc = Oct2Py()

# Call Octave functions from Python
result = oc.sqrt(16.0)  # returns 4.0

# Evaluate Octave code
oc.eval("x = linspace(0, 2*pi, 100);")
y = oc.eval("sin(x)", verbose=False)

# Pass NumPy arrays in and out
import numpy as np
arr = np.array([[1, 2], [3, 4]])
result = oc.det(arr)  # returns -2.0

oc.exit()
CI/CD Without a MATLAB License

If you need to run MATLAB-compatible tests in GitHub Actions or GitLab CI without paying for a MATLAB license, Octave is your only realistic option. Install it via apt install octave in the CI container and run tests with octave --no-gui --eval "run_tests". This works well for pure-numeric computation. It falls apart if your code uses MATLAB-specific toolboxes.

9. Python Replacements — NumPy, SciPy, python-control

NumPy / SciPy / python-control Stack
Free No License Modern Ecosystem
For greenfield projects, Python's scientific stack often replaces MATLAB entirely. NumPy provides N-dimensional arrays and linear algebra; SciPy adds optimization, signal processing (scipy.signal), ODE solvers, and statistics; python-control handles control system design; Matplotlib covers plotting. Performance for vectorized code matches MATLAB for most workloads — the gap has largely closed.
Links: NumPy for MATLAB users · SciPy docs · python-control docs
Pros
  • Completely free — no license, no server
  • Performance matches MATLAB for vectorized code
  • Seamless ML/AI integration (PyTorch, JAX, scikit-learn)
  • Larger talent pool — Python is everywhere
  • Better deployment story — pip, Docker, cloud functions
  • Numba/CuPy for GPU and JIT acceleration
  • Pandas for tabular data, xarray for labeled arrays
Cons
  • No Simulink equivalent for block-diagram modeling
  • MATLAB toolboxes (Control, Image Processing, RF) are more mature
  • 0-indexed arrays — all MATLAB indices must be decremented by 1
  • scipy.signal lacks some specialized MATLAB DSP features
  • Migrating existing MATLAB codebases is significant effort
  • No official MathWorks toolbox-parity guarantee
# MATLAB → Python equivalents

import numpy as np
import scipy.signal as sig
import control

# MATLAB: A = [1 2; 3 4]; eig(A)
A = np.array([[1, 2], [3, 4]])
eigenvalues = np.linalg.eig(A)

# MATLAB: [b,a] = butter(4, 0.2); freqz(b,a)
b, a = sig.butter(4, 0.2)
w, h = sig.freqz(b, a)

# MATLAB: step(tf([1],[1 2 1]))
sys = control.tf([1], [1, 2, 1])
t, y = control.step_response(sys)

# MATLAB: x = linspace(0, 2*pi, 100); (0-indexed in Python)
x = np.linspace(0, 2 * np.pi, 100)
y = np.sin(x)

What the Python stack cannot replace

The hard gaps: Simulink (no equivalent block-diagram simulator), Stateflow, the Embedded Coder (C code generation from MATLAB/Simulink models), and the DO-178C/IEC 61508 certified toolchains used in aerospace and automotive. If your workflow touches any of these, Python is not a drop-in replacement.

10. COM/ActiveX Automation on Windows

win32com.client — Windows COM Automation
Windows Only Legacy License Required
The historical approach before the Engine API existed. Windows COM (Component Object Model) lets Python control the MATLAB application via win32com.client.Dispatch("Matlab.Application"). MATLAB exposes itself as an automation server on Windows. This approach predates the Engine API and is essentially never the right choice for new code — but you may encounter it in legacy Windows automation scripts.
Links: pywin32 on PyPI · COM automation with Python (pbpython)
Pros
  • No MATLAB Engine package required
  • Can interact with already-running MATLAB GUI
  • Works with very old MATLAB versions
  • Familiar pattern if you already automate Excel via COM
Cons
  • Windows only — no macOS, no Linux
  • Strictly worse than the Engine API in every way
  • Full MATLAB license required
  • Fragile: depends on COM registry state
  • No type safety — errors surface at runtime
  • No async mode
  • MathWorks does not document or support this path
# Legacy COM approach — do not use for new projects
import win32com.client

# Start or connect to MATLAB
matlab = win32com.client.Dispatch("Matlab.Application")

# Execute a command string
matlab.Execute("x = sqrt(16)")

# Get a workspace variable
x = matlab.GetVariable("x", "base")  # returns 4.0

# Put a variable into MATLAB workspace
matlab.PutWorkspaceData("myVar", "base", 42.0)

# Prefer matlab.engine instead:
# import matlab.engine; eng = matlab.engine.start_matlab()
Do Not Use for New Projects

The COM/ActiveX approach is Windows-only, undocumented by MathWorks, and strictly inferior to the Engine API in reliability, type safety, and cross-platform support. The only reason to use it is if you are maintaining legacy code that already depends on it and cannot be changed.

11. ZMQ / Socket Bridges — Custom IPC

ZMQ / pymatbridge / Custom Socket Bridges
Advanced License Required Real-Time IPC
Custom inter-process communication bridges using ZeroMQ (ZMQ) allow Python and MATLAB to run as independent processes and exchange data in real time. pymatbridge wraps this into a library, while hand-rolled solutions use REQ-REP or PUB-SUB patterns directly. Useful for long-running MATLAB sessions that receive work items from Python, stream results, or participate in polyglot pipelines (e.g., MATLAB doing signal processing while Python handles I/O and visualization).
Links: pymatbridge GitHub · matlab-zmq MEX bindings · ZeroMQ docs
Pros
  • Persistent MATLAB session — no per-call startup cost
  • Decoupled processes — crash isolation
  • Works across processes, machines, and languages
  • PUB-SUB enables real-time data streaming
  • Not limited by Engine API data size restrictions
Cons
  • Significant implementation complexity
  • Serialization overhead — everything must be encoded/decoded
  • pymatbridge is community-maintained and not actively developed
  • matlab-zmq requires MEX compilation
  • No security: arbitrary MATLAB execution by design
  • Full MATLAB license still required
  • Debugging cross-process issues is painful
# Python side: REQ-REP pattern with ZMQ
import zmq, json, numpy as np

ctx = zmq.Context()
sock = ctx.socket(zmq.REQ)
sock.connect("tcp://localhost:5555")

# Send a computation request
request = {"cmd": "fft", "data": np.random.randn(1024).tolist()}
sock.send_json(request)
response = sock.recv_json()
result = np.array(response["result"])

Decision Guide

Work through these questions in order to identify the right approach for your situation:

1. Do you have a MATLAB license available?
  • No, and I can't get one: Go to Q2.
  • Yes, developer license only (no production deployment): Engine API, subprocess, or kernel approaches. Go to Q4.
  • Yes, including MATLAB Compiler: MCR deployment is an option. Go to Q3.
  • Yes, including Production Server or Web App Server: Enterprise path available. Go to Q5.
2. No MATLAB license — what do you actually need?
  • Numeric computation compatible with MATLAB syntax: Use GNU Octave. Most scripts run unchanged.
  • Greenfield project, not committed to MATLAB syntax: Use NumPy/SciPy/python-control — no license, better ecosystem, better deployment.
  • You have a compiled MCR artifact from someone else: Install the free MCR (matching version) and call it directly.
  • Simulink models are involved: There is no free alternative — you need a MATLAB/Simulink license.
3. You have MATLAB Compiler — do end users need a MATLAB license?
  • Yes, end users need license-free access: Compile with mcc and distribute with the free MCR. Use compiler.build.pythonPackage for Python consumers.
  • End users are internal developers who have MATLAB: MCR is unnecessary overhead. Use the Engine API or subprocess instead.
4. You have a developer MATLAB license — how are you integrating?
  • Python script that calls MATLAB functions: Use the Engine API (matlab.engine). It is the official path.
  • Jupyter notebooks for research or teaching: Use jupyter-matlab-proxy. It is official, maintained, and includes browser desktop access.
  • CI/CD pipeline — run MATLAB tests or scripts: Use subprocess + -batch. Zero Python dependencies, correct exit codes.
  • Long-running MATLAB session processing Python requests: Use a ZMQ bridge if you need real-time streaming, or keep an Engine session open for sequential calls.
  • Windows-only legacy system: COM/ActiveX if you cannot change it, but plan to migrate to the Engine API.
5. You have a Production Server or Web App Server license — who consumes the MATLAB?
  • Other applications via HTTP API (Java, Python, JavaScript, .NET): Deploy compiled functions to MATLAB Production Server. Clients call REST endpoints.
  • Non-technical internal users via a browser: Build an App Designer app, compile it, and host on MATLAB Web App Server.
  • AI agents or chat applications: Use MPS with MCP tool support (available R2024b+).
6. Is startup latency a hard constraint?
  • Cannot tolerate 15–30 second startup: Keep a persistent MATLAB session alive (Engine API with shared session, or a ZMQ bridge). The startup cost is one-time, not per-call.
  • Startup is fine, calls must be fast: Engine API with synchronous calls — each function call takes milliseconds once the engine is running.
  • Completely latency-sensitive real-time: MCR-compiled shared library called inline from C/C++ is the fastest option. No Python overhead.

matlab.engine (Engine API)
Official, well-documented, handles type conversion. Use shared sessions to avoid startup cost. Accept the version pinning constraint.
Distributing to End Users
MATLAB Compiler + MCR
Only option for license-free delivery of MATLAB logic. Painful version pinning, but it works. Consider compiled Python package for Python consumers.
Enterprise REST Service
MATLAB Production Server
Proper scaling, health checks, Prometheus metrics, language-agnostic REST clients. Heavy license cost, but the right tool at enterprise scale.
Internal Dashboard for Non-Engineers
MATLAB Web App Server
Non-technical users get a polished browser interface with no MATLAB install. Requires App Designer skills and a MWAS license.
No License Available
GNU Octave or NumPy/SciPy
Octave for existing MATLAB scripts. NumPy/SciPy for new projects. Octave for toolbox-dependent code only if the toolbox has an Octave Forge equivalent.
Greenfield Numeric Project
NumPy / SciPy / python-control
No license, better deployment, ML/AI integration, larger talent pool. Accept the migration cost from MATLAB idioms. Do not use MATLAB unless Simulink is required.
Real-Time Streaming Pipeline
ZMQ Bridge
Only option when MATLAB must receive continuous input or stream results to Python without per-call startup. Complex to build; justify before committing.
Teaching / Education
jupyter-matlab-proxy on JupyterHub
Students get notebooks in a browser with no local MATLAB install. MathWorks offers academic licensing for institutions. Littlest JupyterHub works for small cohorts.
The Honest Take on MATLAB in 2026

MATLAB's integration story has improved significantly since the Engine API landed on PyPI. For serious engineering workflows that depend on Simulink, specialized toolboxes, or certified code generation, MATLAB remains without peer — and the Engine API or Production Server are solid integration points. For pure numeric computation, Python's scientific stack is now genuinely competitive and offers a far better deployment and ecosystem story. The painful cases are organizations maintaining large MATLAB codebases who want Python interop: the Engine API is the right answer there, with subprocess as a lower-complexity fallback for CI/CD.
