Markdown to ReqIF: A Practical Guide to the Tool Landscape

ReqIF is the lingua franca for exchanging requirements between the automotive, aerospace, and defense supply chain. Markdown is how engineers and developers write everything else. Bridging the two is harder than it looks — here is an honest survey of every tool and approach that exists today.

What is ReqIF and Why Would You Convert from Markdown?

The Requirements Interchange Format (ReqIF) is an XML-based open standard published by the Object Management Group (OMG) and actively promoted by the prostep ivip association. Its latest formal version is ReqIF 1.2 (published July 2016), though the underlying XML schema is unchanged from 1.0.1 — meaning files from any version are binary-compatible. Files use the .reqif extension; when attachments are bundled in a ZIP, the extension becomes .reqifz.

ReqIF dominates wherever requirements must cross organisational boundaries: an OEM sends a system-level requirements package to a Tier-1 supplier who imports it into a different tool, adds derived requirements, and ships it back. The standard is deeply embedded in ASPICE (Automotive SPICE), ISO 26262 (functional safety), and DO-178C (avionics software) workflows. Mainstream commercial tools — IBM DOORS (from v9.5), Siemens Polarion, PTC Codebeamer, Jama Connect, and ReqView — all speak ReqIF natively.

So why would anyone write requirements in Markdown? Several reasons are increasingly common:

  • Teams already using GitHub or GitLab prefer plain-text, diff-friendly authoring.
  • Open-source embedded and aerospace projects (RTEMS, ROS 2 Space) maintain requirements in version control alongside code.
  • Startups and SMEs want to avoid expensive RM tool licences for initial design phases, then need to deliver a ReqIF package to a customer or assessor.
  • AI-generated or LLM-assisted requirement drafts arrive as Markdown and must be ingested into a regulated project.
Bottom line upfront

There is no mature, one-command markdown → .reqif converter. The closest thing is StrictDoc, which treats Markdown as experimental input and can export its internal model to ReqIF. Every other approach requires scripting, intermediate formats, or a commercial tool in the loop.

The Structural Mismatch

To understand why conversion is non-trivial, it helps to appreciate what ReqIF actually encodes. A ReqIF document is not a document in the prose sense — it is a typed, attributed, hierarchically organised, cross-referenced data model. The root is a <REQ-IF> element containing five major sections:

  • DATATYPES — primitive and enumerated type definitions (String, Integer, Boolean, XHTML, Enumeration, Date, Real).
  • SPEC-TYPES — named schemas for objects and relations, each listing which attributes are legal and what data type each carries (SPEC-OBJECT-TYPE, SPEC-RELATION-TYPE, SPECIFICATION-TYPE).
  • SPEC-OBJECTS — the actual requirements. Each object is an instance of a SPEC-OBJECT-TYPE and carries typed attribute values, most importantly ReqIF.Text (rich XHTML), ReqIF.Name (short title), and a machine-readable ReqIF.ForeignID.
  • SPEC-RELATIONS — directed, typed links between objects (e.g., derives, refines, copies).
  • SPECIFICATIONS — one or more named trees of SpecHierarchy nodes, each referencing a SPEC-OBJECT, forming the document outline.

Markdown, by contrast, has a flat or lightly nested structure: headings, paragraphs, lists, and inline formatting. It carries no explicit notion of requirement objects, attribute types, enumerated status fields, unique identifiers, or inter-requirement links. The mismatch is not cosmetic — it is a difference between a document markup language and a requirements data model.

Key challenges in conversion

Identity: Every SPEC-OBJECT needs a globally unique, stable identifier (GUID). Markdown has none.
Type schema: ReqIF requires every object to reference a SPEC-OBJECT-TYPE. Which type do your Markdown headings map to?
Rich text: Markdown formatted text must be serialised as a namespaced XHTML fragment inside a ATTRIBUTE-VALUE-XHTML element.
Attributes: ReqIF consumers expect fields like Status, Priority, and Verification Method. Markdown has none unless you embed front-matter YAML.
Hierarchy vs. flat: Markdown heading levels can approximate SpecHierarchy nesting, but section headings are not the same as requirement objects.

A Minimal ReqIF File

The following abbreviated XML shows the skeleton of a valid ReqIF file containing one requirement. Real files produced by DOORS or Polarion include dozens of data types and attributes; the structure below is deliberately minimal to illustrate the essential scaffolding. 

<?xml version="1.0" encoding="UTF-8"?>
<REQ-IF xmlns="http://www.omg.org/spec/ReqIF/20110401/reqif.xsd"
        xmlns:xhtml="http://www.w3.org/1999/xhtml">

  <THE-HEADER>
    <REQ-IF-HEADER IDENTIFIER="_header_001">
      <CREATION-TIME>2026-03-01T00:00:00.000+00:00</CREATION-TIME>
      <TITLE>Sample Specification</TITLE>
    </REQ-IF-HEADER>
  </THE-HEADER>

  <CORE-CONTENT><REQ-IF-CONTENT>

    <DATATYPES>
      <DATATYPE-DEFINITION-STRING IDENTIFIER="DT-String"
        LONG-NAME="String" MAX-LENGTH="1024"/>
      <DATATYPE-DEFINITION-XHTML  IDENTIFIER="DT-XHTML"
        LONG-NAME="XHTML"/>
    </DATATYPES>

    <SPEC-TYPES>
      <SPEC-OBJECT-TYPE IDENTIFIER="SOT-Requirement"
                        LONG-NAME="Requirement">
        <SPEC-ATTRIBUTES>
          <ATTRIBUTE-DEFINITION-STRING IDENTIFIER="AD-ID"
            LONG-NAME="ReqIF.ForeignID">
            <TYPE><DATATYPE-DEFINITION-STRING-REF>DT-String</DATATYPE-DEFINITION-STRING-REF></TYPE>
          </ATTRIBUTE-DEFINITION-STRING>
          <ATTRIBUTE-DEFINITION-XHTML  IDENTIFIER="AD-Text"
            LONG-NAME="ReqIF.Text">
            <TYPE><DATATYPE-DEFINITION-XHTML-REF>DT-XHTML</DATATYPE-DEFINITION-XHTML-REF></TYPE>
          </ATTRIBUTE-DEFINITION-XHTML>
        </SPEC-ATTRIBUTES>
      </SPEC-OBJECT-TYPE>
    </SPEC-TYPES>

    <SPEC-OBJECTS>
      <SPEC-OBJECT IDENTIFIER="SO-001" LAST-CHANGE="2026-03-01T00:00:00.000+00:00">
        <TYPE><SPEC-OBJECT-TYPE-REF>SOT-Requirement</SPEC-OBJECT-TYPE-REF></TYPE>
        <VALUES>
          <ATTRIBUTE-VALUE-STRING THE-VALUE="SYS-001">
            <DEFINITION><ATTRIBUTE-DEFINITION-STRING-REF>AD-ID</ATTRIBUTE-DEFINITION-STRING-REF></DEFINITION>
          </ATTRIBUTE-VALUE-STRING>
          <ATTRIBUTE-VALUE-XHTML>
            <DEFINITION><ATTRIBUTE-DEFINITION-XHTML-REF>AD-Text</ATTRIBUTE-DEFINITION-XHTML-REF></DEFINITION>
            <THE-VALUE>
              <xhtml:div><xhtml:p>The system <xhtml:strong>shall</xhtml:strong>
                operate within -40°C to +85°C ambient temperature.
              </xhtml:p></xhtml:div>
            </THE-VALUE>
          </ATTRIBUTE-VALUE-XHTML>
        </VALUES>
      </SPEC-OBJECT>
    </SPEC-OBJECTS>

    <SPECIFICATIONS>
      <SPECIFICATION IDENTIFIER="SPEC-001" LONG-NAME="System Requirements">
        <CHILDREN>
          <SPEC-HIERARCHY IDENTIFIER="SH-001">
            <OBJECT><SPEC-OBJECT-REF>SO-001</SPEC-OBJECT-REF></OBJECT>
          </SPEC-HIERARCHY>
        </CHILDREN>
      </SPECIFICATION>
    </SPECIFICATIONS>

  </REQ-IF-CONTENT></CORE-CONTENT>
</REQ-IF>

Notice that the requirement text itself is an XHTML fragment wrapped in a namespace-prefixed xhtml:div. This is the part that makes simple XML generation tricky: you must parse Markdown, convert it to valid XHTML, and embed it inside the ReqIF structure without breaking the namespace declarations. The prostep ivip Implementation Guide (v1.8) lists which XHTML elements tools must support, and mandates that importing tools accept all elements specified in ReqIF §10.8.20.

The Tool Landscape

StrictDoc
Open Source Python Best Option

StrictDoc is the most capable open-source tool in this space. It is a requirements management system built around its own DSL format (SDoc), but it also accepts Markdown files as experimental input and can export its in-memory document model to ReqIF. The project is maintained by Stanislav Pankevich and the strictdoc-project organisation on GitHub.

Version: 0.19.0 (March 2026)  ·  github.com/strictdoc-project/strictdoc  ·  strictdoc.readthedocs.io  ·  License: Apache-2.0

How it works

StrictDoc reads .sdoc and .md files into a unified in-memory document tree. From that tree it can emit HTML, RST, ReqIF, PDF, JSON, or Excel. The ReqIF export is invoked via: 

pip install strictdoc
strictdoc export ./docs --formats=reqif --output-dir ./out

The Markdown path is: .md file → StrictDoc internal model → ReqIF XML. Because Markdown lacks explicit requirement blocks, StrictDoc treats each Markdown document as a document node and any ## Heading as a section node. There are no typed SPEC-OBJECTS with attribute schemas unless you annotate requirements using StrictDoc's own SDoc markup within the Markdown file.

ReqIF import is also supported: a DOORS or Polarion .reqif file can be imported and converted to .sdoc, edited in StrictDoc's web UI, then re-exported to ReqIF — a genuine round-trip workflow. The --reqif-multiline-is-xhtml flag controls whether multi-line text fields are serialised as XHTML or plain STRING attributes.

StrictDoc 2025 brought several improvements to the ReqIF pipeline: better integer-field handling (converted to STRING on import since StrictDoc has no INTEGER type), migration from TOML to Python-based project configuration, and a full rename of the [SECTION] syntax to [[SECTION]].

Strengths
  • Only open-source tool with a full Markdown → ReqIF path
  • Bidirectional ReqIF round-trip (import + export)
  • Active development; regular releases throughout 2025
  • Web UI for editing requirements in-browser
  • Source-code traceability links for ASPICE compliance
  • Companion reqif Python library for lower-level access
Limitations
  • Markdown support is explicitly "experimental"
  • Plain Markdown lacks typed attributes; full feature-set requires SDoc syntax
  • ReqIF output may need tuning per target tool (DOORS vs. Polarion differ)
  • No GUI installer; requires Python environment
  • SPEC-OBJECT-TYPE schema is auto-generated, not manually configured
reqif (Python library)
Open Source Python Low-level

The reqif package — also from the strictdoc-project — is a pure-Python ReqIF parser and serialiser with no opinion about how you generate the data. It gives you an in-memory Python object tree that maps one-to-one onto the ReqIF XML structure. You bring your own Markdown parser and write the glue code.

Version: 0.0.48 (December 31, 2025)  ·  github.com/strictdoc-project/reqif  ·  pypi.org/project/reqif  ·  License: Apache-2.0

Usage pattern for Markdown → ReqIF

import uuid, markdown
from reqif.models.reqif_spec_object import ReqIFSpecObject
# Parse your Markdown with any Python library (e.g. mistune, markdown)
# Build ReqIF model objects manually, then serialise:
from reqif.reqif_bundle import ReqIFBundle
from reqif.unparser import ReqIFUnparser

bundle = ReqIFBundle.create(...)  # assemble from your data
ReqIFUnparser.unparse(bundle, "output.reqif")

The library also ships a CLI for validating, pretty-printing, and anonymising existing .reqif files. The reqif passthrough command is useful for smoke-testing: it parses a file to objects then re-serialises it; the output should be identical to the input.

Strengths
  • Actively maintained; 14 releases through 2025
  • Clean Apache-2.0 licence; suitable for commercial projects
  • Validates against official OMG XSD schema
  • Supports .reqifz compressed bundles
  • Tolerant parser handles malformed inputs from legacy tools
Limitations
  • No Markdown parsing — you must supply this yourself
  • Requires non-trivial boilerplate to assemble a complete bundle
  • Sparse documentation beyond README and examples folder
  • No GUI or CLI for end-to-end Markdown conversion
Doorstop + doorstop-reqif
Open Source Python + Rust Niche

Doorstop stores requirements as YAML files with Markdown text blocks in a Git repository. The separate doorstop-reqif project (by INVAP, an Argentine aerospace company) converts a Doorstop document tree into a ReqIF file. StrictDoc was originally a fork of Doorstop before being rewritten.

Doorstop: github.com/doorstop-dev/doorstop  ·  doorstop-reqif v1.0.0 (April 2024): github.com/invap/doorstop-reqif  ·  License: AGPL-3.0

How it works

Each Doorstop item is a YAML file; the text field is Markdown. doorstop-reqif (written in Rust) walks the Doorstop document tree and serialises each item as a SPEC-OBJECT, rendering the Markdown text as HTML/XHTML for the ReqIF.Text attribute. Links between requirements become SPEC-RELATIONS.

The tool was developed to feed requirements into Capella MBSE tooling. Its roadmap lists "render HTML in ReqIF description from Doorstop item's Markdown" and "export links in requirements" as planned items — indicating the initial release may have only partial Markdown-to-XHTML rendering.

Strengths
  • Real-world aerospace use case (INVAP)
  • Preserves requirement links as SPEC-RELATIONS
  • Doorstop is a mature, git-native RM approach
  • Written in Rust — fast and self-contained binary
Limitations
  • Requires adopting the Doorstop ecosystem (YAML per requirement)
  • 3 stars on GitHub; very limited community
  • AGPL-3.0 licence may restrict commercial use
  • Markdown → XHTML rendering noted as incomplete in roadmap
Pandoc (custom Lua writer)
Open Source Haskell + Lua DIY

Pandoc does not include a ReqIF writer. It is not in pandoc --list-output-formats and no community-maintained Lua writer for ReqIF appears to exist in the pandoc/lua-filters repository. However, Pandoc's custom writer API is capable of generating arbitrary XML from the Pandoc AST.

pandoc.org/custom-writers.html  ·  License: GPL-2.0

What a Lua ReqIF writer would need to do

A custom reqif.lua writer would intercept AST nodes (Header, Para, BulletList, etc.) and emit ReqIF XML. Headings would map to section SPEC-OBJECTS; paragraphs inside sections would become the ReqIF.Text XHTML content. The challenge is that the writer would need to maintain global state for: UUID generation, type registry (SPEC-OBJECT-TYPE), hierarchy stack (SPEC-HIERARCHY nesting), and cross-references. None of this is Pandoc's job — it would all be custom Lua.

This approach is theoretically sound and gives maximum control, but requires substantial development effort for a correctly interoperable output. Anyone starting this project today would be better off building on top of the reqif Python library and calling Pandoc only for the Markdown → XHTML step.

Strengths
  • Pandoc handles all Markdown dialect nuances
  • GFM tables, footnotes, and fenced code blocks are parsed correctly
  • Lua writer can be iterated quickly
Limitations
  • No existing ReqIF writer exists — you build from scratch
  • ReqIF UUID/type management in Lua is awkward
  • Pandoc AST has no concept of requirement objects or UIDs
  • Validation against OMG XSD requires external tooling
pyreqif
Unmaintained Python

An older Python implementation of the ReqIF object model with import and export, plus converters for reqif2html and xlsx2reqif. Last released February 2021 (v0.7). The reqif library from the strictdoc-project has superseded it entirely.

github.com/ebroecker/pyreqif  ·  License: BSD-3-Clause

Not recommended for new projects. Cited here only for completeness; you will encounter references to it in older forum posts and StackOverflow answers.

Commercial RM Tools (DOORS, Polarion, Codebeamer, ReqView)
Commercial Indirect

The major requirements management platforms all speak ReqIF natively but none of them accept Markdown as a direct import format. The typical approach is a two-hop pipeline: Markdown → Word/HTML (via Pandoc) → import into the RM tool → export to ReqIF.

Tool-specific notes

  • IBM DOORS 9.5+ — Full ReqIF 1.2 support with proprietary extensions. Import from Word/Excel is supported; Markdown requires a pre-conversion step. DOORS Next (the cloud version) has richer import options via REST API.
  • Siemens Polarion — Built-in ReqIF import/export with lossless exchange. Supports a "Document Round-trip" with Word for offline editing. A known limitation: Polarion does not support ReqIF references (the same SPEC-OBJECT appearing in multiple Specifications).
  • PTC Codebeamer — ReqIF is a core ALM feature; also supports REST API and MS Office round-trips. In early 2025, a third-party AI-powered tool was demonstrated that ingests unstructured requirements text and generates Codebeamer-compatible ReqIF — the closest thing to "Markdown-ish prose in, ReqIF out" in the commercial ecosystem.
  • ReqView — Lightweight desktop/web tool focused on ReqIF round-trips. Imports from DOORS ReqIF and exports back. No Markdown input.

The DIY Approach: A Practical Recipe

If you have Markdown files with some structure — say, YAML front-matter for metadata and ## headings for requirement groups — the most controllable path today is to write a small Python script using the reqif library:

  1. Parse Markdown with mistune or python-markdown.
  2. Walk the AST: headings become section nodes; paragraphs matching a "shall" pattern become requirement candidates.
  3. Assign deterministic UUIDs (e.g., uuid5(namespace, requirement_id)) so repeated conversions produce stable identifiers.
  4. Render the Markdown body of each requirement to an XHTML string.
  5. Assemble ReqIFSpecObject instances and wrap them in a ReqIFBundle.
  6. Call ReqIFUnparser.unparse(bundle, "output.reqif").
  7. Validate with reqif validate --use-reqif-schema output.reqif.
Tip: YAML front-matter is your attribute store

Add a YAML front-matter block to each Markdown requirement file. Fields like id, status, priority, and verification_method map cleanly onto ATTRIBUTE-DEFINITION entries in your SPEC-OBJECT-TYPE. This is exactly what Doorstop does, and it is the pattern StrictDoc's SDoc format evolved from.

Watch out: tool-specific dialects

The prostep ivip Implementation Guide exists because every tool interprets ReqIF slightly differently. If your ReqIF output needs to round-trip through DOORS, you must use DOORS's expected attribute names (ReqIF.ForeignID, ReqIF.Text, ReqIF.Name, ReqIF.ChapterName) and respect its section-heading expectations. If targeting Polarion, avoid using ReqIF object references across multiple Specifications. Test your output by importing it into the target tool early.

Conclusion

The gap between Markdown's simplicity and ReqIF's structural richness is real and deliberate. Markdown was designed for document authoring; ReqIF was designed for requirements data exchange. The fact that no turnkey Markdown-to-ReqIF converter dominates the market reflects this mismatch — it is not an oversight.

That said, the situation is improving. StrictDoc is the closest thing to a complete open-source solution and is under active development as of March 2026 (v0.19.0). Its native SDoc format closes the structural gap by giving plain-text requirements a proper grammar, typed attributes, UIDs, and parent-child links — all the things ReqIF needs and that Markdown cannot express. If you are starting a requirements project that will eventually need to produce ReqIF, using SDoc (or at minimum Markdown-with-YAML-front-matter inside StrictDoc) is worth the marginal overhead.

The reqif Python library is a reliable foundation for anyone building custom tooling. With 14 releases across 2024–2025 and official OMG schema validation support, it is the most credible low-level building block available.

For teams in regulated industries with an existing RM tool in place, the pragmatic answer remains unchanged: use Pandoc to convert Markdown to Word or HTML, import that into your tool, and let the tool produce the ReqIF. It is a two-step process, but it benefits from the tool's battle-tested ReqIF implementation and preserves the metadata model that your supply-chain partners expect.

Watch this space

The prostep ivip ReqIF Implementor Forum continues to publish interoperability test results, and LLM-assisted requirement extraction (demonstrated with Codebeamer in early 2025) suggests a future where AI-generated Markdown can be enriched with structure and piped to ReqIF with less manual effort. StrictDoc's roadmap also includes improving Markdown support from "experimental" to first-class — that milestone, whenever it arrives, will meaningfully lower the barrier for git-native requirements teams.
