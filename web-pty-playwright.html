Web PTY + Playwright — Automating and Recording Terminal Sessions

A PTY renders a real terminal in the browser. Playwright drives it like a user. Together they let you automate, test, and record CLI sessions visually — with actual colors, fonts, and timing.

The idea in one sentence: Spin up a tool like ttyd to expose a shell in the browser via xterm.js, then use Playwright to type into that terminal — recording the session as a video, GIF, or asciinema cast.

Part 1 — What is a PTY?

A PTY (pseudo terminal) is a kernel-level mechanism that emulates a hardware terminal. It has two ends: a master (held by your terminal emulator) and a slave (which appears to programs like bash as a real terminal device, e.g. /dev/pts/3).

This matters because programs behave differently when they detect a terminal. Bash enables color output, readline, and job control. Git shows progress bars. Programs that call isatty() check whether stdout is a PTY — and change behavior accordingly. Without a PTY, piping output to a script suppresses all of that.

Browser Server ┌──────────────┐ ┌──────────────────────────────┐ │ xterm.js │◄──────────►│ WebSocket ↔ PTY master │ │ (renders │ WS/HTTP │ │ │ terminal, │ │ PTY slave → bash / your app │ │ sends keys)│ └──────────────────────────────┘ └──────────────┘

A web PTY bridges the PTY master's I/O over a WebSocket so a browser-based terminal emulator (xterm.js) can connect to a real shell running on a server.

Part 2 — The Web PTY Tools

xterm.js
JS/TS
The browser-side terminal emulator. Parses ANSI escape sequences, maintains a screen buffer, and renders via Canvas2D or WebGL. Used by VS Code, JupyterLab, GitHub Codespaces, and most web PTY tools as their frontend. The hidden .xterm-helper-textarea element is how keyboard input is captured.
ttyd
C
A single binary that wraps a command in a PTY and serves it over WebSocket + HTTP. Bundles xterm.js internally. Minimal, fast, and the most practical choice for automation. Each WebSocket connection forks a new shell process. Run ttyd bash and you have a browser terminal in seconds.
GoTTY
Go
Same idea as ttyd, written in Go. Default mode is read-only (viewers see output but can't type). Less actively maintained than ttyd but easier to cross-compile. The --permit-write flag enables input.
Playwright
Microsoft
Browser automation for Node.js, Python, Java, .NET. Controls Chromium, Firefox, and WebKit via CDP. Has built-in video recording, WebSocket interception, and keyboard APIs. The piece that drives the terminal and records what it sees.

Part 3 — How Playwright Types Into a Terminal

xterm.js captures keyboard input through a hidden <textarea> element. Playwright sends keyboard events to it exactly like a real user would. The key methods: 

// Focus the terminal first — required before typing
await page.locator('.xterm-helper-textarea').click();

// Type a command with realistic keystroke delay
await page.keyboard.type('ls -la', { delay: 60 }); // 60ms between keys
await page.keyboard.press('Enter');

// Special keys
await page.keyboard.press('Control+C');  // interrupt
await page.keyboard.press('ArrowUp');    // history
await page.keyboard.press('Tab');         // autocomplete

// DON'T use fill() — it bypasses keyboard events entirely
// xterm.js won't see it

Waiting for a command to finish

The hard part: knowing when the shell is done. Two reliable strategies:

// Strategy 1: wait for the prompt to reappear
async function waitForPrompt(page) {
  await page.waitForFunction(() => {
    const rows = [...document.querySelectorAll('.xterm-rows > div')];
    return rows.slice(-3).some(r => /[\$❯#]\s*$/.test(r.textContent));
  }, { timeout: 30000 });
}

// Strategy 2: echo a sentinel value and wait for it
async function run(page, cmd) {
  const sentinel = `__DONE_${Date.now()}__`;
  await page.keyboard.type(`${cmd}; echo "${sentinel}"`);
  await page.keyboard.press('Enter');
  await page.waitForFunction(
    s => document.querySelector('.xterm-rows')?.textContent.includes(s),
    sentinel, { timeout: 60000 }
  );
}

Part 4 — Recording the Session

Option A: Playwright video (WebM)

const context = await browser.newContext({
  recordVideo: { dir: './recordings', size: { width: 900, height: 500 } },
  viewport: { width: 900, height: 500 },
});

// ... run your session ...

await context.close(); // triggers video write
// Video saved to ./recordings/*.webm
// Convert to GIF: ffmpeg -i recording.webm -vf "fps=15,scale=900:-1" out.gif

Option B: asciinema cast (intercept WebSocket frames)

const events = [];
const startTime = Date.now();

page.on('websocket', ws => {
  ws.on('framereceived', frame => {
    // ttyd output frames start with '0'
    if (typeof frame.payload === 'string' && frame.payload[0] === '0') {
      const time = (Date.now() - startTime) / 1000;
      events.push([time, 'o', frame.payload.slice(1)]);
    }
  });
});

// ... run your session ...

// Write .cast file (playable with: asciinema play session.cast)
const header = { version: 2, width: 80, height: 24, timestamp: Math.floor(Date.now()/1000) };
const castContent = [JSON.stringify(header), ...events.map(e => JSON.stringify(e))].join('\n');
fs.writeFileSync('session.cast', castContent);

Part 5 — Full Pipeline

Start ttyd
ttyd -p 7681 bash — serves a real bash shell over WebSocket with xterm.js as the frontend
Launch Playwright with video recording
Open a browser context with recordVideo enabled and navigate to the ttyd URL
Wait for xterm.js to initialize
Wait for .xterm-helper-textarea and click to focus it
Execute commands with realistic timing
page.keyboard.type(cmd, { delay: 60 }) simulates a human typing. Wait for the prompt between each command.
Close context → video is written
Playwright flushes the .webm file. Optionally intercept WebSocket frames during the run to also produce an asciinema .cast
(Optional) Convert to GIF
ffmpeg -i recording.webm -vf "fps=15,scale=900:-1" output.gif

Part 6 — Testing TUI Apps in CI

This is the most practical use case: running E2E tests against terminal UI applications (ncdu, vim, lazygit, htop, your own CLI) in GitHub Actions.

# .github/workflows/test-tui.yml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install ttyd
        run: |
          wget -qO /usr/local/bin/ttyd \
            https://github.com/tsl0922/ttyd/releases/latest/download/ttyd.x86_64
          chmod +x /usr/local/bin/ttyd
      - name: Install Playwright
        run: npx playwright install chromium
      - name: Run TUI tests
        run: npx playwright test
// Test that vim opens and we can quit it
test('vim opens and quits', async ({ page }) => {
  await page.goto('http://localhost:7681');
  await page.waitForSelector('.xterm-helper-textarea');
  await page.locator('.xterm-helper-textarea').click();

  await waitForPrompt(page);
  await page.keyboard.type('vim test.txt\n');
  await page.waitForTimeout(500);

  // Exit vim
  await page.keyboard.press('Escape');
  await page.keyboard.type(':q!');
  await page.keyboard.press('Enter');
  await waitForPrompt(page);
});

vhs — the closest off-the-shelf equivalent

vhs by Charmbracelet is the most mature tool that essentially implements this pattern natively. You write a .tape script, it spins up a headless Chromium with xterm.js internally, runs the commands, and produces a GIF or MP4. Worth knowing about if you don't want to wire this up yourself. 

# demo.tape
Output demo.gif
Set FontSize 14
Set Width 1200
Set Height 600

Type "echo 'Hello from vhs'"
Enter
Sleep 500ms
Type "ls -la"
Enter
Sleep 1s
When to wire it up yourself (Playwright + ttyd) vs. use vhs: Use vhs for producing polished recordings of known commands. Use the Playwright + ttyd combo when you need assertions on output (CI testing), dynamic command generation, or integration with an existing Playwright test suite.
