About llmfit

llmfit is a Rust CLI tool (21k GitHub stars, released February 2026, currently v0.8.9) that detects local hardware and scores every model in its curated database for fit, speed, and quality. It can recommend models, compare them side-by-side, plan hardware upgrade requirements, and download GGUF weights — all from a single static binary with no runtime dependencies.

Key design choices worth noting:

  • No telemetry — nothing about your hardware or queries is sent to any remote server
  • Offline-capable — the core model database is embedded; HuggingFace queries are optional enrichment
  • JSON-first — every subcommand supports --json for tool/agent integration
  • Estimates, not benchmarks — speed (tok/s) and memory figures are heuristic estimates based on quantization math, not measured benchmarks on the specific hardware
  • llama.cpp runtime assumed — all recommendations assume you'll run models via llama.cpp (or a wrapper like Ollama); no PyTorch inference path
How to run it Download the static musl binary (no libc deps), verify the SHA256 checksum, and run directly. Add --no-dashboard if you don't want the web UI to start on port 8787.

curl -sL https://github.com/AlexsJones/llmfit/releases/download/v0.8.9/llmfit-v0.8.9-x86_64-unknown-linux-musl.tar.gz -o llmfit.tar.gz
sha256sum llmfit.tar.gz # verify against .sha256 file
tar -xzf llmfit.tar.gz && ./llmfit system --no-dashboard

The 6.25 GB available RAM (vs 7.72 GB total) reflects OS + WSL2 overhead consuming roughly 1.5 GB before any model is loaded. This available figure is what llmfit uses for fit calculations.

Fit Overview — 792 Models

llmfit's embedded database contains 792 models as of v0.8.9. Every model is scored against the detected hardware. Fit levels:

Fit LevelCountMeaning
Marginal Runnable 472 (60%) Model fits in available RAM and can be loaded. Memory headroom may be tight — OS and other processes compete for the same pool. Inference will be CPU-only and slow.
Too Tight Won't Fit 320 (40%) Model exceeds available RAM at the recommended quantization level. Would require either more RAM, a GPU, or aggressive quantization reduction below what llmfit considers quality-acceptable.

Notably, there are no "Good" or "Perfect" fits in this result set — those levels require a GPU with sufficient VRAM, or a machine with significantly more RAM. Every model that fits on this hardware gets the "Marginal" label, regardless of how much headroom it leaves.

"Marginal" doesn't mean broken Despite the label, "Marginal" models are fully runnable. The term means the model is within the available memory envelope but with less headroom than llmfit considers comfortable. For casual / research use, all 472 Marginal models should load and run. For production use or multi-task workloads where RAM contention is real, the tighter-fitting models may cause OOM conditions.

The MoE Advantage on RAM-Constrained Hardware

The most striking finding from running llmfit is that the top 9 models by score are all Mixture-of-Experts (MoE) architectures. This is not a coincidence — MoE provides a specific advantage when RAM is the limiting constraint.

Why MoE fits better on low-RAM hardware

In a dense model, all parameters are active for every token. A 7B dense model uses all 7B parameters per forward pass, and all parameters must be resident in memory.

In a MoE model, the parameter count is split across N "experts" (feed-forward networks), and only K experts are active per token. The rest sit idle. Since all parameters still need to be in memory (on CPU — GPU expert offloading doesn't apply without a GPU), the RAM savings come from the quantization interaction: a model with 7B parameters but 1B active can be quantized more aggressively overall while maintaining quality at the active-parameter level.

The practical result: MoE models punch above their weight in llmfit's quality score relative to their RAM footprint.

Important caveat: CPU-only MoE is slower than it should be llmfit notes on every MoE model: "MoE architecture, but expert offloading requires a GPU." On a GPU, unused experts can be offloaded to CPU RAM while active experts run on VRAM — this is where MoE truly shines for speed. On CPU-only, all experts are in RAM and the routing overhead is pure cost. The estimated tok/s figures above assume CPU-only execution. Actual inference speed on CPU with MoE models may be lower than a dense model of equivalent active parameter count.

The Memory Boundary

The line between "Marginal" (runnable) and "Too Tight" (won't fit) falls at almost exactly 6.25 GB — the detected available RAM:

  • Largest runnable model: 6.2 GB (ByteDance academic-ds-9B at Q3_K_M)
  • Smallest model that won't fit: 6.4 GB (GigaChat3-10B-A1.8B at Q4_K_M)

The gap is just 200 MB. Several models that don't fit today would fit with a small increase in available RAM — closing other applications, disabling swap, or running on a fresh boot without the ~1.5 GB WSL2 overhead.

llmfit's plan subcommand calculates what hardware changes are needed to run specific models. Output from llmfit plan <model> --context 8192 --json:

A 5.7–7.5 GB GPU (RTX 3060 12GB or similar) would move Llama-3.1-8B from CPU-only to GPU-accelerated, increasing speed roughly 3× at minimum. An RTX 4060 Ti 16GB or RTX 3090 would put it in "Perfect" fit range.

Llama-3.1-70B-Instruct (a large frontier model)

GPU (full offload)46 GB VRAM3.6Dual RTX 4090 or H100 80GB SXM

70B models are firmly out of reach on consumer hardware without specialist GPUs. At 1.1–1.3 tok/s on CPU, a 70B response would take minutes per reply.

The sweet spot for this machine: 1–5B parameter models The practical range for CPU-only inference on 8 GB RAM is 1B–5B parameter models at Q4–Q8 quantization. MoE models can extend this to apparent 6–9B parameter counts via sparse activation. For anything larger, a GPU with ≥8 GB VRAM transforms the situation: an RTX 3060 12GB (~$250 used) would make 8B models run well and 13B models possible.

Tool Observations and Quirks

Running llmfit revealed several behaviors worth noting for anyone using it:

What works well

  • JSON output is consistent and well-structured — --json works on system, fit, recommend, info, diff, and plan
  • Hardware detection is fast — system profiling completes in under a second, no external tool dependencies on Linux
  • The diff subcommand is genuinely useful — side-by-side comparison of two models against the same hardware, including score components and GGUF source links
  • Quantization upscaling — llmfit often recommends a higher quantization than the model default when RAM allows (e.g., Q5_K_M or Q8_0 instead of Q4_K_M). This is a meaningful quality improvement the user might not think to make manually.
  • GGUF source linking — where available, the tool provides direct HuggingFace repo links for community-quantized GGUF files (primarily Unsloth and mradermacher)

Limitations observed

  • search subcommand doesn't support --json — outputs a rich terminal table but piping to JSON fails. Workaround: use fit --json and filter client-side.
  • hf-search returned no results for "instruction" — the live HuggingFace GGUF search may require different search terms or a network path the container couldn't reach. The embedded database is the reliable source.
  • No GGUF sources for several top models — Phi-mini-MoE-instruct and Phi-tiny-MoE-instruct have no listed GGUF sources, meaning you'd need to manually find or convert them. The model recommendations are theoretically sound but not always immediately actionable.
  • Speed estimates are heuristic — tok/s figures are calculated from quantization math and hardware specs, not measured. Real-world CPU inference on a i7-8700K will be heavily influenced by memory bandwidth, OS scheduling, and thermal throttling in a way the model can't capture. Treat estimates as order-of-magnitude guidance.
  • "Marginal" applies to all CPU-only results regardless of headroom — a model using 30% of RAM gets the same "Marginal" label as one using 95%. The fit score components provide more nuance, but the fit level itself loses meaning in CPU-only scenarios.
  • No score for 216 Qwen3 variants — the search results showed many models with score: -, indicating they weren't fully evaluated (likely MLX-format or non-GGUF variants that llama.cpp can't run).

Score component breakdown

llmfit's composite score has four components, each 0–100:

ComponentWhat it measuresTypical values (CPU-only 8GB)
FitHow much memory headroom the model leaves86–100 for models using <90% of RAM
SpeedEstimated tokens/second relative to usability threshold100 for most CPU models (heuristic ceiling)
ContextContext window length adequacy100 for most models (4k+ is sufficient for basic use)
QualityModel quality relative to database-wide range58–73 for small models on this hardware

On CPU-only hardware, speed and context scores are frequently 100 (the models that fit can run fast enough and have adequate context), so the quality component dominates differentiation between models. A higher-quality model that uses more RAM will score higher than a lower-quality model that fits more comfortably.

Conclusions

What actually runs on 8 GB RAM with no GPU

More than expected. 472 of 792 models in the llmfit database technically fit. For practical use, the 1B–5B parameter range offers the best experience: fast enough for interactive use (~40–100 tok/s estimated), small enough to leave OS headroom, and the quantization quality is high enough to be useful.

MoE is the architecture for RAM-constrained hardware

Every top recommendation is a MoE model. The apparent parameter count (7B, 8B, 9B) is misleading — the active parameters per token are 1–2B, matching the quality of a smaller dense model at much higher apparent sophistication. On CPU without GPU offloading, the speed advantage of MoE is reduced, but the RAM efficiency remains.

The GPU gap is real but not enormous for modest models

A used RTX 3060 12GB (~$250) would change this machine from "CPU-only marginal" to "GPU-accelerated Good fit" for 8B models. The performance improvement would be roughly 3–10× in tokens/second. For anyone serious about local LLM inference, GPU + moderate RAM is a significantly better investment than more RAM alone.

llmfit is a useful tool with real rough edges

The JSON API is well-designed for agent and tool integration. The hardware planning output is genuinely actionable. The limitations (heuristic speeds, missing GGUF sources for some top models, search subcommand not JSON-compatible) are real but don't undermine the core use case: quickly understanding what's possible on a given machine without manually calculating quantization math.

Recommended starting point for this hardware

  • Best overall: LiquidAI/LFM2-8B-A1B via Unsloth's GGUF — 128k context, solid quality, available GGUF
  • Fastest response time: microsoft/Phi-tiny-MoE-instruct at Q8_0 — 97 tok/s estimated, only 2.1 GB
  • Most RAM efficient + open source: meta-llama/Llama-3.2-1B-Instruct — 1.8 GB, Apache 2.0, widely tested
  • Embeddings for RAG: nomic-ai/nomic-embed-text-v1.5 — 137M parameters, 0.2 GB, 450 tok/s
  • Code: Qwen/Qwen2.5-Coder-1.5B-Instruct — 2.5 GB, strong code model for its size
