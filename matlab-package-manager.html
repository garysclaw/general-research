MATLAB Package Management — What Actually Exists and How It Works

There are at least four separate "package managers" for MATLAB. They don't all do the same thing, they don't talk to each other, and MathWorks is only now catching up to what pip did in 2011.

The short version: MATLAB's package management story is fragmented and historically neglected. The OS-level mpm tool installs MATLAB itself in headless environments. A separate in-MATLAB mpm system (R2024b+) manages community packages. The community built tbxmanager years ago because nothing official existed. And .mltbx files are the distribution format. None of these fully connect.

The Four Tools — What Each One Actually Does

OS-level mpm — the installer tool
Official
A standalone binary from MathWorks. Downloads and installs MATLAB itself and official MathWorks toolboxes from the command line. This is for Docker, CI/CD, and enterprise provisioning — not for managing third-party packages. You run it outside MATLAB, not inside it.
In-MATLAB mpm — the new package system (R2024b+)
New in R2024b
A completely separate system from the above. Runs inside MATLAB. Uses mpminstall, mpmsearch, mpmcreate, and a mpackage.json manifest. This is MathWorks' attempt at something like pip or npm — but the ecosystem is nearly empty and it's missing a lockfile. Does not talk to the Add-On Explorer or .mltbx system.
Add-On Explorer — the GUI
Official
The point-and-click interface in the MATLAB desktop. Browses MathWorks licensed toolboxes and the File Exchange. Installs .mltbx files. Fine for interactive use, useless for scripted or headless workflows without workarounds.
tbxmanager — community-built package manager
Community
A single .m file with its own package registry. Built years ago because nothing official existed. Popular in control systems and optimization (YALMIP, MPT3, SeDuMi). Doesn't support versioned pinning. Still actively used in academic circles because it hosts the specific toolboxes those communities need.

OS-Level mpm: Installing MATLAB in Headless Environments

This is the tool you use when setting up MATLAB in Docker, CI/CD, or on a fresh server. Download the binary and run it outside MATLAB.

# Download mpm (Linux)
wget -q https://www.mathworks.com/mpm/glnxa64/mpm
chmod +x mpm

# Install MATLAB R2025b with toolboxes
./mpm install \
  --release=R2025b \
  --destination=/opt/matlab/R2025b \
  --products=MATLAB \
              Signal_Processing_Toolbox \
              Statistics_and_Machine_Learning_Toolbox \
              Deep_Learning_Toolbox

# Pin to a specific update level
./mpm install --release=R2024bU3 \
  --destination=/opt/matlab --products=MATLAB

# Download for offline use first, install later
./mpm download --release=R2025b \
  --destination=/path/to/cache --products=MATLAB

./mpm install --source=/path/to/cache \
  --destination=/opt/matlab --products=MATLAB

Product names use underscores. Common ones: MATLAB, Simulink, Deep_Learning_Toolbox, Statistics_and_Machine_Learning_Toolbox, Parallel_Computing_Toolbox, Signal_Processing_Toolbox, Control_System_Toolbox, Optimization_Toolbox.

Docker workflow

FROM ubuntu:22.04

RUN wget -q https://www.mathworks.com/mpm/glnxa64/mpm && \
    chmod +x mpm && \
    ./mpm install \
      --release=R2025b \
      --destination=/opt/matlab \
      --products=MATLAB Signal_Processing_Toolbox && \
    rm -f mpm

ENV MLM_LICENSE_TOKEN=<your-batch-token>
ENTRYPOINT ["matlab", "-batch"]

GitHub Actions

- name: Set up MATLAB
  uses: matlab-actions/setup-matlab@v2
  with:
    release: R2025b
    products: Simulink Deep_Learning_Toolbox

- name: Run tests
  uses: matlab-actions/run-tests@v2
The licensing problem: Unlike pip or npm, you can't run headless MATLAB in CI without a paid license token (Batch Licensing Token). There's no free tier for automated use. This is the single biggest blocker for CI/CD adoption — it's an institutional procurement problem before it's a technical one.

In-MATLAB mpm (R2024b+): Managing Community Packages

This is MathWorks' attempt at a real package manager that runs inside MATLAB. It uses a mpackage.json manifest and semantic versioning. The tooling exists; the ecosystem doesn't yet.

% Search for packages
mpmsearch("signal processing")

% Install a package
mpminstall("some-package-name")

% Install a specific version
spec = matlab.mpm.PackageSpecifier("mypackage", Version="1.2.3");
mpminstall(spec)

% Install from local folder (must have mpackage.json)
mpminstall("/path/to/my/package")

% Install in editable mode (live changes reflected immediately)
mpminstall("/path/to/my/package", Editable=true)

% List installed packages
mpmlist

% Uninstall
mpmuninstall("mypackage")

% Add a private registry (e.g. Artifactory)
mpmAddRepository("https://registry.example.com/matlab")

The mpackage.json manifest

{
  "schemaVersion": "1.1.0",
  "name": "my-package",
  "version": "1.0.0",
  "description": "Signal analysis utilities",
  "matlabVersion": ">=24.2.0"
}

MATLAB release versions map to semver: R2024b = 24.2.0, R2025a = 25.1.0. Run mpmcreate to scaffold a new package — it generates the manifest automatically.

Known gap: Namespace folders (those starting with +) are not supported as package roots in the new mpm system. This means MATLAB's only real code isolation mechanism and the new package manager are mutually incompatible. That's a significant design problem.

Installing .mltbx Files Programmatically

The .mltbx format is the older, more widely-used distribution format. Under the hood it's just a ZIP archive. You can install these without touching the GUI.

% Install a local .mltbx file
matlab.addons.install('/path/to/mytoolbox.mltbx')

% Install without license prompt
matlab.addons.toolbox.installToolbox('/path/to/mytoolbox.mltbx', true)

% Download from File Exchange and install
websave('/tmp/tb.mltbx', 'https://www.mathworks.com/matlabcentral/mlc-downloads/...');
matlab.addons.install('/tmp/tb.mltbx')

% List all installed add-ons
matlab.addons.installedAddons()

% Uninstall by name
matlab.addons.uninstall('My Toolbox Name')

tbxmanager: The Community Alternative

Bootstrap it once, then use it to install toolboxes from its own curated registry. Popular in control systems and optimization research — YALMIP, MPT3, SeDuMi, Casadi interfaces all live here.

% Bootstrap (run once)
urlwrite('http://www.tbxmanager.com/tbxmanager.m', 'tbxmanager.m');
tbxmanager savepath

% Add to startup.m for persistence
tbxmanager restorepath

% Install packages
tbxmanager install yalmip
tbxmanager install mpt sedumi       % multiple at once

% Update
tbxmanager update
tbxmanager update yalmip

% Browse
tbxmanager show available
tbxmanager show installed

The Honest Comparison

Feature pip npm MATLAB mpm (R2024b+)
Central public registry PyPI npmjs.com None (File Exchange ≠ registry)
Lockfile for reproducibility Poetry/uv: yes package-lock.json No
Semantic versioning Yes Yes R2024b+ only
Dependency resolution Yes Yes Partial
Namespace / env isolation venv node_modules Global path only
Free headless/CI use Yes Yes Requires paid license token
Community package count Millions Millions Nascent

Real-World Workaround: Vendor Dependencies in Git

Because package management is unreliable for reproducibility, most serious MATLAB projects still do this:

project/
├── src/
│   └── my_analysis.m
├── vendor/               ← dependencies committed directly to the repo
│   ├── jsonlab/
│   └── export_fig/
└── setup.m               ← adds vendor paths on startup
% setup.m
addpath(genpath(fullfile(fileparts(mfilename('fullpath')), 'vendor')));

Verbose and unsophisticated compared to a requirements.txt, but it works and requires nothing beyond Git. It's what the community lands on when the official tools fall short.

Creating and Packaging Your Own Toolbox

1
Structure your code
Keep source in a toolbox/ folder, examples in examples/, docs in doc/. Use +namespace folders for scoping if needed (but note the new mpm system can't handle namespace-rooted packages).
2
Package via GUI or programmatically
GUI: Home > Add-Ons > Package Toolbox. Programmatic: use matlab.addons.toolbox.ToolboxOptions and matlab.addons.toolbox.packageToolbox — better for CI pipelines.
3
The GUID matters
Every toolbox has a GUID in its .prj file. MATLAB uses this to track installed versions and upgrade in place. If the GUID changes, it becomes a different toolbox to the installer.
4
Distribute via File Exchange for discoverability
Publishing your .mltbx on MATLAB File Exchange and linking your GitHub repo makes it appear in the Add-On Explorer automatically. That's the closest MATLAB has to a public registry right now.
% Programmatic packaging
opts = matlab.addons.toolbox.ToolboxOptions("/path/to/root", "your-guid-here");
opts.ToolboxName = "My Analysis Toolbox";
opts.ToolboxVersion = "2.1.0";
opts.OutputFile = "/path/to/release/MyToolbox.mltbx";
matlab.addons.toolbox.packageToolbox(opts)
Bottom line: Use the OS-level mpm for provisioning MATLAB installations. Use matlab.addons.install for scripted .mltbx installs. Use tbxmanager if you're in control/optimization and need YALMIP or MPT. Use the new in-MATLAB mpm only if you're building new packages and want to follow the modern standard — but expect the ecosystem to be thin for a few more years.
